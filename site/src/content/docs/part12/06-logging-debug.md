## このページでできるようになること

- ログレベル（error/warn/info/debug/trace）を目的で使い分けられる
- esp-printlnのログがどう設定されているかを説明できる
- printデバッグの限界と、defmt/probe-rsという選択肢の存在を知る

## 先に結論

この教材のexamplesはすべて`log`クレート＋`esp-println`でログを出しています。ログは「とりあえずprintln」ではなく、**レベル**（重要度）を付けて出すのが基本です。レベルの選別はビルド時に決まるため、リリース版では詳細ログを削って容量と実行時間を節約できます。ただしシリアルへのログ出力は実行タイミングを変える「観測の副作用」を持つことも知っておく必要があります。その先には、バイナリ形式で軽量にログを送るdefmtと、デバッガプローブとして働くprobe-rsという選択肢があります（紹介のみ。本教材の必修ではありません）。

## 身近なたとえ

ログレベルは保健室の記録に似ています。「骨折の疑い（error）」「熱が高め（warn）」「今日の来室者数（info）」「一人ひとりの体温メモ（debug）」を全部同じ声の大きさで報告したら、大事な知らせが埋もれてしまいます。重要度で仕分けしてあれば、普段はinfoまで、調査のときだけdebugまで聞く、という切り替えができます。

たとえと違うのは、マイコンでは「どこまで聞くか」を**ビルドするときに**決めるという点です。あとから現場で聞き直すことはできません。

## 仕組み

### ログレベルの使い分け

`log`クレートには5つのレベルがあります。最終プロジェクトの実際の使い分けを添えます。

| レベル | 目的 | final-wireless-buttonでの例 |
|---|---|---|
| `error!` | 機能が失われた。放置できない | （教材コードでは未使用。パニック相当はesp-backtraceが担当） |
| `warn!` | 異常だが動き続けられる | ACKが返らない、解読できないパケットを受信 |
| `info!` | 通常運転の節目 | 起動完了、ボタン押下、エラー状態からの復帰 |
| `debug!` | 開発中だけ見たい詳細 | ハートビート毎回の送信、ACK受信のたびの表示 |
| `trace!` | さらに細かい追跡 | （未使用） |

「毎回出るもの」をinfoにすると流量が多すぎて節目が埋もれます。500ms周期のハートビートがdebugなのはそのためです。

### esp-printlnの仕組みと設定

examplesのログは次の2つで動いています（この構成はcargo check済みのexamples群と同じですが、このページ自体に検証済みの新規コードはありません）。

- 各プログラムの冒頭で`esp_println::logger::init_logger_from_env()`を呼ぶ
- 出力レベルは**ビルド時の環境変数`ESP_LOG`**で決まる。本リポジトリではexamples/.cargo/config.tomlに`ESP_LOG = "info"`が設定してある

つまり`debug!`のログが見えないのは故障ではなく、フィルタで落ちているだけです。`ESP_LOG=debug`にしてビルドし直せば見えます。逆に言うと、**実行中にレベルを切り替えることはできません**（レベルの選別がビルド時に確定するため）。

```bash
# 一時的にdebugレベルでビルド・実行する例
ESP_LOG=debug cargo run --release -p final-wireless-button --bin receiver
```

### printデバッグの限界

ログは最強の入門デバッグ手段ですが、限界があります。

1. **タイミングが変わる** — シリアル出力には時間がかかるため、ログを足した途端にチャタリングやレースの再現が変わることがあります（観測が現象を変える）
2. **出せない場所がある** — 割り込みの奥深くやパニックの最中など、出力自体が難しい文脈があります（パニック時の表示はesp-backtraceが担ってくれています）
3. **量が増えると読めない** — だからレベル分けと、`[送信]` `[受信]`のような**モジュール名プレフィックス**（final-wireless-buttonで実践しています）が効きます

### その先の選択肢 — defmtとprobe-rs

名前だけでも知っておくと、次のステップで迷いません。

- **defmt** — ログの文字列整形をホストPC側で行う仕組みです。マイコンは小さなIDとバイナリデータだけを送るため、出力が軽く、タイミングへの影響が小さくなります
- **probe-rs** — Rust製の書き込み・デバッグツールです。ESP32-C6は**USB Serial/JTAGを内蔵**（GPIO12/13、専用ハード）しているので、追加のデバッガ装置なしでブレークポイントやステップ実行に挑戦できる素地があります

本教材ではespflash＋logの構成で通していますが、「ログでは追えない」壁に当たったら、この2つを調べてみてください。

## よくある失敗

1. **`debug!`のログが出ない！と慌てる** — `ESP_LOG`のレベルで落ちているだけ、が大半です。ビルド時設定なので、環境変数を変えたら**ビルドし直し**が必要です
2. **全部`info!`で出す** — 節目のログが周期ログに埋もれます。「毎回出るものは`debug!`」を目安に
3. **ログを足したらバグが消えた（Heisenbug）** — 出力の時間で実行タイミングが変わった可能性があります。ログ位置を減らす・変える、ログに頼らずLEDや状態変数で観測するなどの工夫を

## やってみよう

examples/final-wireless-buttonを`ESP_LOG=debug`でビルド・実行して、`info`のときには見えなかったログ（ハートビート毎回の送信、ACK受信）がどれだけ増えるか観察してみましょう。増えたログの中で「常時見えるべきだ」と思うものがあれば、レベルを変える根拠を説明できるか考えてみてください。

## 確認問題

1. 500ms周期で毎回出るログに適したレベルはどれですか。理由も述べてください。
2. `ESP_LOG=debug cargo run ...`としてもdebugログが出ないとき、まず確認すべきことは何ですか。
3. defmtが「軽い」のはなぜですか。

<details>
<summary>答え</summary>

1. `debug!`です。通常運転の節目（info）が周期ログに埋もれないようにするためです。
2. ビルドし直されたかどうかです。`ESP_LOG`はビルド時に評価されるため、キャッシュされたビルドのままでは反映されません（環境変数の変更で再ビルドされるはずですが、意図した値でビルドされたかの確認が先です）。
3. 文字列の整形をホストPC側で行い、マイコンからはIDとバイナリデータだけを送るからです。送る量と整形の処理時間が減り、実行タイミングへの影響が小さくなります。

</details>

## まとめ

- ログはレベルで仕分ける。「毎回出るものはdebug、節目はinfo、異常はwarn/error」
- esp-printlnのレベルはビルド時の`ESP_LOG`で決まる。実行中には変えられない
- ログは観測の副作用を持つ。限界を感じたらdefmt/probe-rsという次の道具がある
