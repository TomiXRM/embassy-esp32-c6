## このページでできるようになること

- 周辺機器の初期化失敗を`Option`で受け止め、機能を絞って動き続ける「劣化運転」を設計できる
- panicハンドラの中で`try_lock`を使う理由を説明できる
- 周期の違う複数の仕事（20ms/5ms/10ms/4ms）を1つのファームウェアに共存させる構造を説明できる
- KeyballをC6で作り直すときに変わる部分・変わらない部分を仕分けできる

## 先に結論

実物のキーボードはキースキャンだけではありません。Keyballにはトラックボール（光学センサPMW3360）、OLED表示、LEDがあり、それぞれ**動く周期も、壊れ方も違います**。読み解くと3つの設計が見えてきます。第一に、トラックボールの初期化は`init().await.ok()`で失敗を握りつぶさず`Option<Ball>`に変換し、**ボール無しでもキーボードとして動き続ける**——第12部7ページで学んだ縮退運転（degraded operation。このページの題材まわりでは劣化運転とも呼びます）の実戦形です。第二に、OLEDはMutexで共有しつつ、**panicハンドラからだけはtry_lock**で触る——止まれない場面でロック待ちをしない技法です。第三に、スキャン20ms・ボール5ms・USBキーボード10ms・マウス4msという**バラバラの周期は、周期ごとに独立のフューチャーへ分け、Channelでつなぐ**ことで自然に共存します。どれも教材で学んだ部品の組み合わせであり、新しい魔法はありません。

## 身近なたとえ

飲食店の厨房を想像してください。揚げ物のフライヤーが故障した日でも、店は「揚げ物以外のメニュー」で営業を続けます（劣化運転）。伝票を通す注文システムは全員で共有しますが、火災報知器が鳴った緊急時には「システムが空いていれば記録する、ふさがっていれば諦めて避難する」と決めておきます（try_lock）。そして炒め物・煮込み・炊飯はそれぞれ固有のリズムで進み、料理人は互いの完成を待たずに自分の持ち場を回します（周期の違う仕事の共存）。

たとえと違うのは、ソフトウェアではこれらの方針が`Option`型・`try_lock`・フューチャー分割という**型と構造**で表現され、方針からの逸脱がコンパイルエラーや型の不一致として機械的に見つかることです。

## 仕組み

### 劣化運転 — Option<Ball>という設計判断

PMW3360はSPI接続の光学センサで、Keyballは自作のasyncドライバで扱っています（RP2040のSPI0＋DMA、7MHz）。初期化はかなり重い仕事です。電源投入後、**SROMと呼ばれるセンサ用ファームウェアをSPI経由でアップロード**し、データシートが指定する待ち時間を`Timer::after_micros`で几帳面に守りながら設定を流し込みます。動き出した後は、移動量などの12バイトを一括で読むburst_readで毎周期データを取ります。

注目すべきは初期化の失敗の扱いです。設計を疑似コードで書くとこうなります（実装の転載ではなく、記事の設計を教材の書き方で表現したものです）。

```rust
// 初期化の失敗(Result)を「有るか無いか」(Option)に変換する
let mut ball = Ball::init(spi).await.ok(); // Option<Ball>

loop {
    // ボールが有るときだけ読む。無くてもループは回り続ける
    if let Some(ball) = ball.as_mut() {
        if let Ok(motion) = ball.read().await {
            // マウスレポートを組み立ててChannelへ
        }
    }
    // キースキャンやレポート送信はボールの有無と無関係に動く
}
```

`ok()`の一行に設計判断が凝縮されています。`Result`を`Option`に変えるとは、「失敗の理由はもう問わない。**以後は『装備が有るか無いか』という状態として扱う**」という宣言です。トラックボールの配線が外れていても、センサが故障していても、キーボードとしての本業は続く。第12部7ページの「全停止せず、できる仕事を続ける」が、`Option<Ball>`という型ひとつで実現されています。

`unwrap()`と比べてみてください。`unwrap()`なら、ボールの故障がキーボード全体の起動失敗になります。周辺機器が増えるほど「全員が健康でないと起動しない」ファームウェアは壊れやすくなる。**どの失敗が全体を止めてよく、どの失敗は絞った運転で吸収すべきか**——この仕分けが統合設計の中心です。

### OLEDの共有と、panicハンドラのtry_lock

OLEDには複数の場所から書きたくなります。通常の表示更新に加えて、Keyballは**panic時のメッセージ表示**にもOLEDを使います。そこで画面は`Mutex`（第9部9ページ)で包んで共有します。実際の型は`Mutex<ThreadModeRawMutex, Option<Oled>>`——ここにも`Option`がいます。OLEDも「無ければ無いで動く」装備なのです。

面白いのはpanicハンドラからのアクセスです。通常のコードは`lock().await`で順番を待てますが、panicハンドラは**awaitできません**。panicはasyncの世界の外で起きる同期的な事件で、そのときexecutorはもう仕事を続けられないからです。さらに、panicを起こしたのが「OLEDのロックを握っていた本人」だった場合、ロック解放を待てば**永遠に待ち続けます**。だからpanicハンドラは`try_lock`を使います——**空いていれば画面にエラーを出す。ふさがっていれば潔く諦める**。最後のメッセージ表示は「できたら嬉しい」機能であって、そのために固まってはいけないのです。

第12部6ページで「panic時の情報をどう残すか」を学びました。実物はそれをOLEDでやり、しかも共有リソースの作法（諦められる場面ではtry_lock）まで守っている、という好例です。

### 周期の違う仕事の共存

Keyballの中では、少なくともこれだけの周期が同時に流れています。

| 仕事 | 周期 | 周期の根拠 |
|---|---|---|
| キーマトリクススキャン | 20ms | 人間の打鍵に十分な速さ＋消費電力とのバランス |
| トラックボール読み取り | 5ms | ポインタの滑らかさは更新頻度に直結する |
| USBキーボードレポート | 10ms | ホストへ申告したポーリング間隔 |
| USBマウスレポート | 4ms | 同上（遅延に敏感なので短い） |

これを1本のループに押し込むとどうなるでしょう。「4msごとに回して、5回に1回スキャンして……」と最小公倍数のカウンタ管理が始まり、どれか1つの処理が長引くと全員が遅れます。Keyballの答えは教材第9部の答えと同じです。**周期ごとに独立のフューチャーに分け、間をChannelでつなぐ**。スキャンは20msで回って結果をイベントとして流し、ボールは5msで回ってマウスレポートを流し、レポート送信側は自分のペースでChannelから取り出して送る。互いの遅れが波及しにくく、周期の変更も1か所で済みます。

このページの部品がすべて、2ページ目で見た「1つのtaskの中のjoin/selectツリー」の枝としてぶら下がっている、というのが全体像です。

### おまけ: 統合ならではの小さな設計

- **オートマウスレイヤ**: ボールを動かすとマウス用レイヤ（クリックキーなどが並ぶ面)が自動で有効になり、**500ms動きがなければ**元に戻ります。第5ページのレイヤ機構と、タイムスタンプ（`Instant`）の組み合わせです
- **スクロールの余り持ち越し**: ボールの移動量を割り算してスクロール量に変換すると端数が出ます。端数を捨てるとゆっくりした操作が全部消えてしまうので、**余りを次回へ持ち越して蓄積**します。整数だけで小数の効果を得る、組み込みらしい固定小数点的テクニックです

### C6で作るなら — 変わるもの、変わらないもの

第2ページから読み解いてきた内容の総決算として、C6移植の仕分け表を作れます。

| 部品 | C6での扱い |
|---|---|
| 劣化運転（Option化）、Mutex共有、try_lock、周期分割とChannel | **そのまま使える**（embassy-sync/time/futuresのみに依存する設計だから） |
| Tap-Hold・レイヤ・オートマウスの状態機械 | そのまま使える（HAL非依存の純粋ロジック） |
| PMW3360ドライバ | 移植可。SPIは汎用の**SPI2**を使う（C6の汎用SPIは1系統） |
| OLED（I2C） | 移植可（esp-halのI2Cはasync対応） |
| WS2812（LED） | RP2040はPIOで駆動、**C6はRMTペリフェラル**（esp-hal-smartled系クレート）で等価 |
| USBレポート送信 | **BLE（Bluetooth Low Energy）のnotifyに差し替え**（前ページ） |
| 左右通信（PIO 1-wire） | 2線UARTまたはESP-NOWで再設計（6ページ） |
| master/slave判定（USB列挙の早い者勝ち） | USBが無いので不成立。**配線・設定値・起動時の取り決めなど別方式を設計** |

上半分——設計の骨格——がほぼ無傷で移植できることに注目してください。第12部8ページで学んだ「ハードウェア非依存の層を厚くする」設計は、テストのためだけでなく、**チップを乗り換える日のための保険**でもあったのです。

## よくある失敗

- **初期化を全部unwrapして「全員健康でないと起動しない」ファームウェアにする** — 周辺機器が1つ外れているだけで全機能が死にます。本業に必須でない装備は`Option`化して劣化運転に落とし込みましょう
- **panicハンドラで普通のlockを取る** — panicした本人がロック保持者だった場合に固まります。panicハンドラ・割り込みなど「待てない文脈」では`try_lock`で諦められる設計にします
- **周期の違う仕事を1つのループのカウンタで管理する** — 最小公倍数の管理が複雑化し、1つの遅延が全体に波及します。周期ごとにフューチャーを分け、Channelでつなぎましょう
- **Noneの経路を一度も試さない** — `Option<Ball>`にしても、ボール無しで起動するテストをしなければ劣化運転は絵に描いた餅です。装備を外して起動してみる、が最良の確認です

## やってみよう

教材の最終プロジェクト（examples/final-wireless-button）の受信側に「OLEDが後から追加される」と想像して、(1) OLED初期化が失敗してもLED表示だけで動き続ける、(2) panic時にはOLEDが空いていればエラーを出す——この2つを満たす型と関数構成を紙に設計してみてください。`Mutex<CriticalSectionRawMutex, Option<Display>>`のような型が自然に出てくれば、このページは身についています。

## 確認問題

1. `Ball::init(spi).await.ok()`の`.ok()`は何を捨て、何を残していますか。この変換が設計として適切なのはどんな条件のときですか。
2. panicハンドラで`lock().await`が使えない理由と、仮に同期版のブロックするlockがあったとしても使うべきでない理由を、分けて説明してください。
3. 周期20msのスキャンと周期4msのマウスレポートを1つのループにまとめた場合に起きる問題を2つ挙げてください。

<details>
<summary>答え</summary>

1. 失敗の理由（エラーの中身）を捨て、「装備が有るか無いか」という情報だけを残しています。その装備が本業に必須でなく、失敗理由ごとに対処を変える必要がない（どう失敗しても「無し」として運転を続ける）場合に適切です。
2. panicハンドラは非同期文脈の外にあり`await`できないため`lock().await`は書けません。ブロックするlockが仮にあっても、panicを起こした当人がロック保持者ならば解放されることはなく永久に待ち続けるため、`try_lock`で「取れなければ諦める」以外に安全な選択はありません。
3. 例: 4msに合わせてループを回しつつカウンタでスキャンを間引く管理が複雑になり周期変更に弱くなる。どれか1つの処理（SPI待ちなど）が長引くと無関係なレポート送信まで一緒に遅れる。ほかに「周期の根拠が1か所に混ざり読めなくなる」も可です。

</details>

## まとめ

- 必須でない装備は`Result`を`Option`へ畳んで劣化運転（縮退運転）に落とす。「どの失敗が全体を止めてよいか」の仕分けが統合設計の中心
- 共有リソースはMutexで守り、待てない文脈（panicハンドラ）だけはtry_lockで「諦められる」ようにする
- 周期の違う仕事は周期ごとにフューチャーへ分けChannelでつなぐ。HAL非依存に設計された骨格は、チップを乗り換えても無傷で持っていける
