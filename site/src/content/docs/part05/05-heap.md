## このページでできるようになること

- ヒープが何で、スタックと何が違うかを説明できる
- 組み込みでヒープを避ける2つの理由（断片化・確保失敗）を説明できる
- `heapless::Vec` のような固定容量の道具の使い方と考え方が分かる
- それでもヒープを使う例外（無線スタック＋esp-alloc）があることを知る

## 先に結論

ヒープは「好きな大きさを、好きなタイミングで借りられる」メモリの貸出窓口です。便利ですが、マイコンでは**断片化**（空きが細切れになって大きな塊が借りられなくなる）と**確保失敗**（借りられなかったときの行き場のなさ）が致命傷になります。そこで本教材では、必要な最大量を最初から決めておく**固定長設計**を基本にし、`heapless` クレートの `Vec` や `String` を使います。例外は無線スタックで、Wi-FiやBLEを使うときだけ `esp-alloc` でヒープを用意します。

## 身近なたとえ

ヒープは、図書館の自由席のようなものです。空いていれば何席でも借りられますが、混んでくると「1人分の席はあちこちに空いているのに、6人並びの席はどこにもない」状態になります。これが断片化です。

ただし実際のヒープで深刻なのは、席が取れなかったときの扱いです。図書館なら待てば済みますが、24時間動き続けるマイコンでは「確保失敗でプログラム停止」が製品の故障そのものになります。

## 仕組み

### スタックとヒープの違い

| | スタック | ヒープ |
|---|---|---|
| 大きさの決まり方 | コンパイル時に決まる | 実行時に自由 |
| 返し方 | 関数を抜けると自動 | 所有者がいなくなったとき（Rustでは自動だが、タイミングは実行時） |
| 速さ | とても速い | 空きを探す分だけ遅く、時間も一定でない |
| 失敗 | あふれ（前ページ） | 確保失敗がいつでも起こりうる |

`std` の `Vec` や `String` はヒープの上に作られています。`#![no_std]` でそれらが使えなかったのは、ヒープの貸出窓口（アロケータ）が存在しないからです。

### 理由1: 断片化

確保と解放を長時間繰り返すと、空き領域が細切れになります。合計では十分空いていても、**連続した**大きな塊が取れなくなるのです。数日、数か月と動き続ける組み込み機器では、「3週間後のある日突然、確保に失敗する」という再現の難しい故障になります。

```mermaid
graph TB
  subgraph after["長時間動作後のヒープ"]
    direction LR
    u1["使用中"] --- f1["空き4KB"] --- u2["使用中"] --- f2["空き2KB"] --- u3["使用中"] --- f3["空き4KB"]
  end
```

この状態では、空き合計は10KBあっても「連続8KB」の要求は失敗します。

### 理由2: 失敗したときの行き場がない

パソコンなら、メモリ不足のときOSがプログラムを終了させ、ユーザーがアプリを再起動できます。マイコンでは確保失敗はpanic（強制停止）につながり、再起動するまで機器全体が沈黙します。「失敗するかもしれない操作を、いつでも起こりうる形で抱え込む」こと自体が、設計として不利なのです。

### 代わりの道具: heapless

`heapless` クレートは、`Vec` や `String` と似た使い勝手のまま、**容量を型で固定した**コンテナを提供します。データ本体はスタックやstatic領域に置かれ、ヒープを使いません。

次のコードは書き方を示す例です。完全な利用例は第8部以降のexamplesで登場します。

```rust
use heapless::Vec;

// 容量8のVec。8を超えて入れることは絶対にできない
let mut readings: Vec<u16, 8> = Vec::new();

// pushはResultを返す。満杯なら Err で教えてくれる
if readings.push(1023).is_err() {
    // 満杯だったときの対応をここで決める（古い値を捨てる、など）
}
```

ポイントは2つです。

- 容量 `8` が型の一部なので、最大メモリ使用量がコンパイル時に確定する。断片化も確保失敗も原理的に起きない
- `push` が `Result` を返すので、「満杯だったらどうするか」を書く場所が必ずある。第3部で学んだResultの考え方がそのまま活きます

### 例外: 無線スタックとesp-alloc

Wi-FiやBLEの通信スタック（esp-radio）は内部でヒープを必要とします。そのため第10部・第11部のexamplesでは、`esp-alloc` クレートでヒープ（アロケータ）を用意します。これは「無線という複雑な仕事のために、決まった大きさのヒープを起動時に1回だけ確保する」限定的な使い方で、アプリ側のデータ設計は引き続き固定長が基本です。

## よくある失敗

- **`Vec` が使えないので `heapless::Vec` に置き換えたが、容量を考えずに小さくした** — `push` の `Err` を無視すると、データが黙って捨てられます。容量は「起こりうる最大量」から決め、あふれたときの方針（捨てる・上書き・待つ）を必ず書いてください。
- **文字列連結のつもりで `String` を探してしまう** — `heapless::String<N>` はありますが、ログ表示が目的なら `info!("value = {}", x)` のようにフォーマットごと `log` に任せるほうが簡単で、バッファ設計も不要です。

## やってみよう

上の `readings` の例で、容量を `2` にして3回 `push` したら何が起きるか、コードを書かずに紙の上で追ってみてください。3回目の `push` が返すものと、そのとき `readings` に入っている値を答えられたら合格です。

## 確認問題

1. 断片化とは何ですか。「合計」と「連続」という言葉を使って説明してください。
2. `heapless::Vec<u16, 8>` の `8` はどの時点で決まりますか。それがなぜ組み込みで有利ですか。
3. 本教材でヒープを使う例外はどんな場面ですか。

<details>
<summary>答え</summary>

1. 確保と解放の繰り返しで空きメモリが細切れになり、空きの合計は足りているのに連続した塊が取れなくなる現象です。
2. コンパイル時です。実行前に最大メモリ使用量が確定するので、長時間動作しても断片化や確保失敗が起きません。
3. Wi-FiやBLEなど無線スタック（esp-radio）を使う場面です。esp-allocで起動時にヒープを用意します。

</details>

## まとめ

- ヒープは断片化と確保失敗という時限爆弾を抱える。長時間動く機器では固定長設計が基本
- `heapless` は容量を型に持つコンテナ。満杯は `Result` で必ず知らされる
- 無線スタックだけは例外的にesp-allocでヒープを使う
