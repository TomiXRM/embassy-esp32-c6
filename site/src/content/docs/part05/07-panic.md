## このページでできるようになること

- panicが「回復をあきらめた強制停止」であることを説明できる
- no_stdでpanicハンドラを自分で用意する必要がある理由と、esp-backtraceの役割が分かる
- `unwrap` とpanicの関係を理解し、unwrapを使ってよい場面を判断できる

## 先に結論

panic（パニック）は、プログラムが「この状態では正しく続行できない」と判断したときの強制停止です。配列の範囲外アクセスや、`None` に対する `unwrap` で発生します。パソコンではOSがエラー表示を引き受けますが、`no_std` では**panic時に何をするかを自分で用意する義務**があります。本教材ではその係を `esp-backtrace` に任せています。panicが起きるとエラーの発生場所への道筋（バックトレース）をシリアルに表示し、そこで停止します。24時間動く機器にとってpanicは避けたい事態なので、`unwrap` は「失敗が設計上ありえない場所」に限って使います。

## 身近なたとえ

panicは、工場の非常停止ボタンに似ています。おかしな状態のまま機械を動かし続けると被害が広がるので、まず全部止める。これは「壊れたまま走り続けない」ための安全装置です。

ただし工場と違って、マイコンには非常停止ボタンを押したあと様子を見に来る人がいないかもしれません。だからこそ「そもそも押させない設計」（ResultやOptionでの事前処理）が大切になります。

## 仕組み

### panicはいつ起きるか

- `Option` が `None` のときの `.unwrap()` / `Result` が `Err` のときの `.unwrap()`
- 配列の範囲外アクセス（`buf[10]` で `buf` の長さが8、など）
- 明示的な `panic!("メッセージ")`
- 前ページで見た `StaticCell::init` の2回目の呼び出し

共通するのは、「続行したらもっとおかしなことが起きる」とライブラリや言語が判断した場面だということです。第3部で学んだとおり、Rustは失敗の多くを `Result` として**値で**返させます。panicはそこからこぼれた、最後の受け皿です。

### no_stdではpanicの後始末も自分の仕事

`std` のあるパソコンでは、panicするとエラーメッセージの表示とプログラムの終了をOSと `std` がやってくれます。`no_std` にはそれがないため、**panicハンドラ**（panic時に呼ばれる関数)を必ず1つ用意しないと、そもそもビルドが通りません。

blinkyにあったこの1行が、その義務を果たしています。

```rust
use esp_backtrace as _;
```

`as _` は「名前は使わないが、クレートをリンクに含める」という書き方です。esp-backtraceはpanicハンドラを提供し、panicが起きると次の仕事をします。

1. panicのメッセージ（何が起きたか）をシリアルへ表示する
2. **バックトレース**（どの関数がどの関数を呼んだ末にpanicへ至ったかの道筋）を表示する
3. プログラムを停止させる

バックトレースは事故現場に残った足跡です。「どこで転んだか」だけでなく「どの道を通ってそこへ来たか」まで分かるので、原因調査が大幅に楽になります。

### unwrapとどう付き合うか

`unwrap` は「失敗していたらpanicしてよい」という宣言です。禁止ではありませんが、使ってよい場所を選びます。

| 場面 | 判断 | 理由 |
|---|---|---|
| 起動時の初期化（ペリフェラルの取得など） | unwrap可 | 失敗するのは配線や設定の誤りで、続行に意味がない。早く大きな音で失敗するほうが良い |
| 通信の受信データの解析 | unwrap不可 | 相手やノイズ次第で失敗は**正常に**起こりうる。Resultで処理して続行する |
| センサ値の読み取り | 原則不可 | 一時的な失敗はリトライや既定値で回復できる（第12部7ページ） |

目安は「その失敗は、直せるバグか、それとも起こりうる出来事か」です。バグならpanicで早く知らせる。出来事なら `Result` で受け止めて動き続ける。

## 実行方法

わざとpanicを起こして、esp-backtraceの表示を見てみましょう。blinkyの `loop` の直前に次の2行を足して書き込みます。

```rust
let nothing: Option<u32> = None;
let _value = nothing.unwrap(); // ここでpanicする
```

```bash
cargo run --release
```

シリアルモニタに、panicのメッセージと、発生場所への道筋が表示されて停止します。確認したら2行を消して、元に戻してください。

## よくある失敗

- **panicしたのに気づかない** — panic後のマイコンは停止しているだけで、外から見ると「LEDが点かなくなった」だけです。開発中はシリアルモニタをつないだままにして、panicメッセージを見逃さないでください。
- **とりあえず全部unwrapで書いてしまう** — 練習中は動きますが、通信やセンサが絡むと「たまに全停止する機器」ができあがります。unwrapを書くときは「これが失敗するのはバグのときだけか？」と自問し、そうでなければ `match` や `if let`、`?` で処理してください。
- **`use esp_backtrace as _;` を未使用と思って消す** — 名前を使っていないので不要に見えますが、消すとpanicハンドラがなくなりビルドエラーになります。

## やってみよう

上の「実行方法」のpanic実験で、`None` の代わりに `panic!("テスト用のpanicです");` を使う版も試して、メッセージが表示されることを確認してください。終わったら必ず元に戻しましょう。

## 確認問題

1. panicと `Result::Err` の使い分けを一言で説明してください。
2. no_stdでpanicハンドラが必須なのはなぜですか。
3. 起動時の初期化でunwrapが許されるのはなぜですか。

<details>
<summary>答え</summary>

1. 起こりうる出来事は `Err` で値として返して処理を続け、続行不能なバグ状態はpanicで止めます。
2. パソコンではOSと `std` が担っているpanic時の後始末（表示と停止）が、no_stdには存在しないからです。用意しないとビルドできません。
3. 初期化の失敗は配線や設定の誤り（＝直すべきバグ）であり、続行しても意味がないためです。早く目立つ形で失敗するほうが原因を見つけやすくなります。

</details>

## まとめ

- panicは「続行をあきらめる」最後の安全装置。unwrap・範囲外アクセスなどで発生する
- no_stdではpanicハンドラが必須。本教材ではesp-backtraceが表示と停止を担当する
- unwrapは「失敗＝バグ」の場所に限定し、起こりうる失敗はResultで受け止める
