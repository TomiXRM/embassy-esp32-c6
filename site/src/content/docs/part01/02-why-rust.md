## このページでできるようになること

- Arduinoの`loop`一枚岩が苦しくなる典型的な場面を説明できる
- RustとEmbassyに移ると何が変わるか（何が変わらないか）を説明できる
- 「Rustは難しい」の中身が何なのかを知る

## 先に結論

Arduinoが悪いわけではありません。Lチカやセンサ1個なら、Arduinoは今でも最速・最簡単の道具です。苦しくなるのは「LEDを点滅**しながら**、ボタンに**すぐ**反応**しながら**、無線も待つ」といった**同時進行**が増えたときです。`delay()`はプログラム全体を止めてしまうからです。RustとEmbassyでは、仕事を**task**という単位に分割し、待ち時間に他のtaskへ切り替えることで、この問題を素直に書けます。さらにRustのコンパイラは、データの取り違えなど多くのバグを**実行前に**見つけてくれます。

## 身近なたとえ

Arduinoの`loop`は「店員が1人だけの店」です。ラーメンを3分待つ間（`delay`）、店員はタイマーの前で突っ立っていて、レジにお客さんが来ても気づきません。Embassyは「待ち時間に別の仕事へ移る店員」です。麺のタイマーをセットしたらレジへ行き、鳴ったら麺に戻ります。

たとえと違うのは、店員（CPU）は本当に1人のままだということです。人を増やす（CPUを増やす）のではなく、**待ち方を変える**のがEmbassyのやり方です。

## Arduinoではどう書くか — どこで苦しくなるか

LEDを1秒間隔で点滅させつつ、ボタンが押されたらすぐ反応したい、というよくある例です。

```cpp
// Arduino（C++）の素朴な書き方
void loop() {
  digitalWrite(LED_PIN, HIGH);
  delay(1000);              // ← この1秒間、ボタンは無視される
  digitalWrite(LED_PIN, LOW);
  delay(1000);              // ← ここでも無視される
  if (digitalRead(BTN_PIN) == LOW) {
    // ここに来られるのは2秒に1回だけ
  }
}
```

`delay(1000)`の間、プログラムは完全に止まります。ボタンの反応は最悪2秒遅れます。

Arduino経験者なら「`millis()`で時刻を覚えて、delayを使わなければいい」と知っているでしょう。その通りです。ただしその方式は、機能が3つ4つと増えるたびに「前回時刻の変数」と「状態フラグ」が増殖し、`loop`が状態管理だらけの一枚岩になっていきます。動くけれど、読めない・直せないコードになりやすいのです。

## RustとEmbassyではどう書くか

Embassyでは「LED係」と「ボタン係」を別々のtask（タスク）として書きます。次のコードは考え方を示す抜粋で、このままでは動きません（動く完全な形は第9部で扱います）。

```rust
// 概念を示す抜粋です。完全なコードは第9部と examples/06-embassy-tasks を見てください
#[embassy_executor::task]
async fn led_task(mut led: Output<'static>) {
    loop {
        led.toggle();
        Timer::after(Duration::from_millis(1000)).await; // 待つ間、CPUは他のtaskへ
    }
}

#[embassy_executor::task]
async fn button_task(mut button: Input<'static>) {
    loop {
        button.wait_for_falling_edge().await; // 押されるまで眠って待つ
        info!("ボタンが押されました");        // 押された瞬間に反応
    }
}
```

ポイントは2つです。

- **LEDの1秒待ちが、ボタンの反応を邪魔しない**。`await`で待っている間、CPUは他のtaskを進めます
- それぞれのtaskは、単機能の素直な`loop`のまま。機能が増えたらtaskを増やすだけで、一枚岩になりません

## Rustがコンパイル時に守ってくれること

Embassyとは別に、Rust言語そのものの利点があります。Rustのコンパイラは「このデータを2か所から同時に書き換えていないか」「もう手放したデータを使っていないか」を**ビルドの段階で**検査します（所有権と借用。第3部で学びます)。C++で実行してみるまで分からなかった種類のバグの多くが、コンパイルエラーとして先に見つかります。

正直な注意も2つ書いておきます。

- Rustにすれば**バグがゼロになるわけではありません**。ロジックの間違いや設計ミスは普通に起きます
- 最初はコンパイルエラーの多さに面食らいます。ただしエラーは敵ではなく「実行前に見つかったバグの報告書」です。この教材では、エラーが出るたびに「なぜこの制約があるのか」を説明します

## よくある失敗

- **「Arduinoは卒業すべき低レベルな道具」と考えてしまう**: 違います。単機能の工作ならArduinoの方が早いことも多いです。道具の使い分けの問題です
- **「asyncにすれば速くなる」と期待する**: Embassyが解決するのは「待ち時間の使い方」であって、計算そのものは速くなりません。CPUをずっと使う重い計算には別の設計が必要です（第9部で扱います）

## やってみよう

自分が過去に書いた（または見た）Arduinoスケッチをひとつ思い出し、「`delay`中に反応できなくなっているもの」がないか探してみてください。それがこのページで説明した一枚岩の限界の実例です。

## 確認問題

1. `delay(1000)`の何が問題になるのですか。
2. Embassyのtaskに分割すると、なぜボタンの反応が遅れなくなるのですか。
3. 「Rustに移ればバグはなくなる」は正しいですか。

<details>
<summary>答え</summary>

1. その1秒間、プログラム全体が停止し、ボタン入力など他の仕事に一切反応できなくなることです。
2. `await`で待っている間、CPUが他のtaskの処理へ切り替わるからです。LED係の待ち時間中も、ボタン係は入力を待ち受けられます。
3. 正しくありません。所有権などの仕組みでデータの取り違え系のバグはコンパイル時に多く防げますが、ロジックの誤りや設計ミスは残ります。

</details>

## まとめ

- Arduinoの限界は「同時進行」。`delay`が全体を止め、回避策は一枚岩化を招く
- Embassyは仕事をtaskに分割し、待ち時間に他のtaskを進める
- Rustはデータ取り違え系のバグを実行前に検出する。ただし万能ではない
