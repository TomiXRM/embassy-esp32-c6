## このページでできるようになること

- I2Cで起こる代表的なエラーと、その物理的な原因を説明できる
- `AcknowledgeCheckFailed`（NACK）と他のエラーを`match`で書き分けられる
- 「エラーの意味は文脈で変わる」ことを踏まえた対処を設計できる

## 先に結論

I2Cのエラーは`Result`の`Err`として返り、esp-halでは`esp_hal::i2c::master::Error`という`enum`で種類が分かります。最重要は`AcknowledgeCheckFailed`（NACK＝相手が応答しなかった）で、**同じNACKでも文脈によって意味が違います**。バススキャン中なら「不在」という正常な結果、センサ測定中なら「センサが外れた・まだ測定中」という異常です。また、esp-halは長さ0の書き込みを`ZeroLengthInvalid`で拒否します。エラーは`match`で種類ごとに分け、「無視してよいもの」「再試行するもの」「警告するもの」を書き分けます。

## 身近なたとえ

電話をかけて誰も出ないとき、その意味は文脈で変わります。間違い電話の確認なら「誰も出ない＝この番号は使われていない」でむしろ収穫ですが、火事の通報で消防署が出なければ大問題です。「出ない」という事実は同じでも、対処はまったく違います。

ただしプログラムでは「文脈による意味の違い」を人間の勘ではなく、**コードのどの場所でどの`match`腕に落ちたか**で明示的に書き分ける、という点がたとえとの違いです。

## 仕組み

I2Cの代表的なエラーと原因を整理します。

| エラー | 物理的に起きたこと | よくある原因 |
|---|---|---|
| `AcknowledgeCheckFailed` | アドレスまたはデータの後のACKクロックでSDAがLowにならなかった | デバイス不在／アドレス間違い／測定中で応答不可／配線が外れた |
| `ZeroLengthInvalid` | （通信前にドライバが拒否） | 長さ0のバッファでwrite/readを呼んだ |
| その他（タイムアウト・バス異常など） | バスの電気的状態が想定外 | プルアップ不足／SDAとSCLの入れ替わり／ノイズ |

`AcknowledgeCheckFailed`が通信「後」の失敗であるのに対し、`ZeroLengthInvalid`は通信「前」にドライバが弾く点が違います。前者は配線を疑い、後者はコードを疑います。

エラー処理の設計は3択で考えると整理できます。

1. **無視する** — そのエラーが正常な結果の一部であるとき（スキャン中のNACK）
2. **再試行する** — 一時的な失敗の可能性があるとき（測定中のNACK→次の周回でやり直し）
3. **報告して縮退する** — 続行不能なとき（センサ不在→センサ機能なしで動き続ける）

「即パニックで停止」が選択肢にないことに注意してください。組み込み機器は現場で誰も再起動してくれません。

## RustとEmbassyではどう書くか

`examples/04-i2c`には3種類の対処がすべて登場します。これは抜粋です。完全なコードは `examples/04-i2c` を見てください。

まず**スキャン中のNACKは「無視」**です。

```rust
use esp_hal::i2c::master::{Config as I2cConfig, Error as I2cError, I2c};

match i2c.write_async(addr, &[0x00]).await {
    Ok(()) => info!("  デバイスを発見: 0x{:02X}", addr),
    // ACKが返らない = そのアドレスにデバイスはいない（正常なこと）
    Err(I2cError::AcknowledgeCheckFailed(_)) => {}
    // それ以外はバス自体の異常（配線ミス・プルアップ不足など）
    Err(e) => warn!("  0x{:02X} でバスエラー: {:?}", addr, e),
}
```

**測定中の失敗は「再試行」**（次の周回へ受け流し）です。

```rust
if let Err(e) = i2c.write_async(SHT30_ADDR, &[0x2C, 0x06]).await {
    error!("測定コマンドの送信に失敗: {:?}", e);
    continue; // この周回は諦め、2秒後にまた試す
}
```

**センサ不在は「報告して縮退」**です。

```rust
if !sht30_found {
    warn!("SHT30 (0x{:02X}) が見つかりません。配線を確認してください", SHT30_ADDR);
    loop {
        Timer::after(Duration::from_secs(60)).await; // 止まらず待機し続ける
    }
}
```

## コードを一行ずつ読む

```rust
Err(I2cError::AcknowledgeCheckFailed(_)) => {}
```

- `enum`のバリアントをパターンで特定し、**このエラーだけ**を握りつぶします。`Err(_) => {}`と書いてすべてを握りつぶすのとは大違いです。配線ミス（バスエラー）まで沈黙してしまうと、原因調査の手がかりが消えます
- `(_)`の中身はNACKが起きた位置などの詳細情報です。ここでは使わないので`_`で受けます

```rust
Err(e) => warn!("  0x{:02X} でバスエラー: {:?}", addr, e),
```

- `match`は上から順に試されるので、この腕には「NACK以外のエラー」だけが届きます。網羅性チェックのおかげで、エラーの取りこぼしはコンパイル時に検出されます

```rust
error!("測定コマンドの送信に失敗: {:?}", e);
continue;
```

- ログには`{:?}`でエラーの中身を残します。「失敗しました」だけのログは、現場で最も役に立たないログです

## 実行方法

配線は前ページと同じです。動かしたまま、わざとエラーを起こして観察しましょう。

```bash
cd examples/04-i2c
cargo run --release
```

実行中にSHT30のSDA線を抜くと、次の周回から読み出しが失敗し始めます。

```text
INFO - 温度: 26.3 C / 湿度: 58.2 %RH
ERROR - 測定コマンドの送信に失敗: AcknowledgeCheckFailed(...)
ERROR - 測定コマンドの送信に失敗: AcknowledgeCheckFailed(...)
```

線を戻すと測定が復活します。パニックせず、自力で回復するのが良い設計です。

## よくある失敗

- **`Err(_) => {}`で全エラーを無視**: NACK以外の異常（プルアップ不足など）まで見えなくなり、「動かないが何も出ない」最悪のデバッグ状況を自作してしまいます。無視するバリアントは名指しにします
- **エラーで`unwrap`/`expect`を使う**: センサはケーブル1本で「いなくなる」部品です。実行中に起こりうる失敗でパニックすると、装置全体が止まります。`unwrap`は初期化直後など失敗が設計上ありえない箇所に限定します
- **測定中のNACKを「不在」と即断**: センサが前のコマンドを処理中でNACKを返すこともあります。1回の失敗で不在と決めず、再試行してから判断します
- **ログにエラー内容を書かない**: `{:?}`を付けるだけでバリアント名が残り、原因の切り分けが一気に楽になります

## やってみよう

スキャンの`match`から`Err(I2cError::AcknowledgeCheckFailed(_)) => {}`の腕を一時的に削除して、コンパイルは通るがログが100件以上のNACK警告で埋まることを確かめてみましょう（`Err(e)`の腕がNACKも受け取るためです)。「どのエラーを無視するか」の設計がログの品質を決めることが体感できます。終わったら元に戻してください。

## 確認問題

1. 同じ`AcknowledgeCheckFailed`でも、バススキャン中とセンサ読み出し中で対処が違うのはなぜですか。
2. `ZeroLengthInvalid`は他のエラーと発生のタイミングが違います。どう違いますか。
3. エラー処理の3つの基本方針を挙げてください。

<details>
<summary>答え</summary>

1. エラーの意味が文脈で変わるからです。スキャン中のNACKは「不在確認」という正常な結果ですが、読み出し中のNACKは「いるはずのセンサが応答しない」異常で、再試行や警告が必要です。
2. 他のエラーはバス上の通信の結果として起きますが、`ZeroLengthInvalid`は通信を始める前にドライバがコードの誤り（長さ0のバッファ）を拒否したものです。配線ではなくコードを直します。
3. 無視する（正常な結果の一部）、再試行する（一時的な失敗）、報告して縮退する（続行不能）。パニックで止めるのは最後の手段です。

</details>

## まとめ

- I2Cのエラーは`enum`で種類が分かる。`match`で名指しして「無視・再試行・縮退」を書き分ける
- NACKの意味は文脈次第。スキャンでは正常、読み出しでは異常
- `Err(_) => {}`の全無視と実行中の`unwrap`は避け、ログには必ずエラー内容を残す
