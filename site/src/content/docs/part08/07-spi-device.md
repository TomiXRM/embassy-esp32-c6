## このページでできるようになること

- CSのLow区間を「1回の取引」として設計できる
- 全二重転送を使って「コマンド送信＋応答受信」を1つのバッファで行う考え方が分かる
- 実デバイスをつなぐ前にデータシートで確認すべき項目を挙げられる

## 先に結論

実際のSPIデバイスとの通信は、**CSをLowにしている区間を1回の「取引」**として組み立てます。取引の中身は「コマンドのバイト列を送り、続けて応答のバイト数だけクロックを回して受け取る」が基本形です。SPIは全二重なので、応答を受け取るには**ダミーバイト（中身に意味のない送信データ）を送ってクロックを回す**必要があります。デバイスごとの違い（コマンド体系・モード・最大クロック・補助ピン）はすべてデータシートに書いてあり、I2Cセンサのときと同じく「データシートをコードに翻訳する」のがドライバの仕事です。

## 身近なたとえ

自動販売機との取引に似ています。お金を入れてボタンを押す（コマンド送信）ところまでは自分の動作ですが、商品を受け取るには取り出し口に手を入れる（クロックを回して受信する）必要があります。商品は勝手に飛んできません。

ただしSPIでは「手を入れる」＝「ダミーバイトを送る」であり、**受け取りたいバイト数とちょうど同じ数だけ**送らないと応答が途中で切れる、という点がたとえとの違いです。スレーブは自分からクロックを回せないからです。

## 仕組み

たとえば「レジスタ0x0Fの値を1バイト読む」取引は、多くのデバイスで次の形になります。

```text
CS ‾‾‾\____________________________________/‾‾‾
        バイト0        バイト1
MOSI:  [コマンド 0x8F] [ダミー 0x00]
MISO:  [不定（捨てる）] [応答: レジスタの値]
```

- **バイト0**: マスタがコマンドを送っている間、デバイスはまだ応答を用意できないので、MISOに流れてくる値は捨てます
- **バイト1**: ダミーバイトを送ってクロックを回すと、その裏でデバイスが応答を返します
- 転送バッファは`[コマンド, 0x00]`で作り、転送後の`[捨てる, 応答]`から2バイト目だけを使う。これが`transfer_in_place`系と相性のよい組み立て方です

コマンドの値（この例の0x8F＝「読み出しフラグ＋レジスタ番号」のような体系）はデバイスごとに違います。接続前にデータシートで確認する項目をまとめます。

| 確認項目 | 見る場所 | 教材の例（ループバック時の設定） |
|---|---|---|
| SPIモード | タイミング図（CPOL/CPHA） | モード0 |
| 最大クロック | 電気的特性 | まず1MHzから |
| コマンド体系 | レジスタマップ／コマンド表 | デバイス次第 |
| CSの極性 | ピン説明（ほぼ全部Low有効） | Low有効 |
| 補助ピン | ピン説明（ディスプレイのD/C、RESETなど） | デバイス次第 |
| 電源電圧 | 絶対最大定格 | **3.3V品を選ぶ** |

小型SPIディスプレイのように、SPIの4本に加えてD/C（データ/コマンド切り替え）やRESETといった**補助ピンをGPIOで別途操作する**デバイスも多くあります。この場合もCSと同じく、ただの`Output`として自分で制御します。

## RustとEmbassyではどう書くか

取引の骨格は前ページのループバックとまったく同じです。これは抜粋です。完全なコードは `examples/05-spi` を見てください。

```rust
// CS（チップセレクト）は自分でGPIOを操作する方式。
// 通常時はHigh（非選択）にしておき、通信の間だけLowにする
let mut cs = Output::new(peripherals.GPIO21, Level::High, OutputConfig::default());

let mut buf = TX_DATA; // 実デバイスなら [コマンド, ダミー, ...] を入れる

cs.set_low(); // 通信開始（スレーブを選択）
let result = spi.transfer_in_place_async(&mut buf).await;
cs.set_high(); // 通信終了（選択を解除）
```

実デバイスに応用するときに変わるのは、次の3点だけです。

1. `buf`の中身を「コマンド＋ダミーバイト」というデバイスの決まりに合わせる
2. 転送後の`buf`から、応答の位置のバイトだけを取り出して解釈する
3. `SpiConfig`のモードと周波数をデータシートの値に合わせる

転送そのものの書き方（CSで挟む、`transfer_in_place_async`を`await`する、`Result`を`match`する）は一切変わりません。

## コードを一行ずつ読む

```rust
cs.set_low();
let result = spi.transfer_in_place_async(&mut buf).await;
cs.set_high();
```

- **`await`をCSの区間に挟む形**に注目してください。転送が完了してから`set_high()`が実行されるので、CSのLow区間が転送全体を確実に覆います
- 取引を2回に分けたいとき（コマンド転送→少し処理→続きを読む）でも、多くのデバイスは**CSをLowに保ったまま**でないと取引が途切れます。`set_high()`はどこに置くべきか、データシートのタイミング図で確認する癖をつけましょう

```rust
match result {
    Ok(()) => { /* bufの応答バイトを解釈する */ }
    Err(e) => error!("SPI転送エラー: {:?}", e),
}
```

- 転送の`Ok`は「クロックを回し終えた」ことしか意味しません。**デバイスが正しい応答を返したかは別問題**です。CSを繋ぎ忘れても`Ok`は返ります。応答の中身（IDレジスタの期待値など）で検証するのが実デバイス流です

## 実行方法

手元のSPIデバイスに合わせて配線し、`examples/05-spi`の設定（モード・周波数）をデータシートの値に直してから実行します。デバイスがまだ手元にない場合は、前ページのループバックのまま`buf`の先頭を「コマンドのつもりのバイト」にして、転送後のバッファ位置の対応関係を観察するだけでも取引の感覚がつかめます。

```bash
cd examples/05-spi
cargo run --release
```

## よくある失敗

- **応答が全部0xFFまたは0x00**: CSが繋がっていない・Lowになっていない典型症状です。デバイスが選択されておらず、MISOを駆動していません。転送自体は`Ok`になるのが罠です
- **ダミーバイトを送らずに応答を待つ**: スレーブはクロックがないと1ビットも返せません。「読みたいバイト数ぶんダミーを送る」を忘れると応答が受け取れません
- **最初のバイトを応答として解釈する**: コマンド送信中のMISOは不定です。応答はコマンドの「後ろの」バイト位置に現れます
- **5V品のデバイスを直結**: GPIOの絶対最大は3.6Vです。5VロジックのデバイスはC6を壊す恐れがあります。3.3V対応品を選びます

## やってみよう

ループバック配線のまま、`TX_DATA`を`[0x8F, 0x00, ...]`のような「コマンド＋ダミー」風の並びに変えて実行し、受信バッファのどの位置に何が現れるかをログで確認してみましょう。ループバックでは送った値がそのまま返るので、「バイト位置の対応」を安全に観察できます。

## 確認問題

1. SPIで応答を受け取るために「ダミーバイト」を送るのはなぜですか。
2. 転送が`Ok`で終わったのに応答が全部0xFFでした。まず何を疑いますか。
3. 実デバイスをつなぐ前にデータシートで確認すべき項目を3つ挙げてください。

<details>
<summary>答え</summary>

1. クロックを回せるのはマスタだけで、スレーブはクロック1発につき1ビットしか返せないからです。受け取りたいバイト数と同じだけ送信（ダミー）してクロックを回します。
2. CSの配線と制御です。デバイスが選択されていないとMISOは駆動されず、転送自体は成功扱いのまま無意味な値が返ります。
3. 例: SPIモード（CPOL/CPHA）、最大クロック周波数、コマンド体系（レジスタマップ）。ほかにCS極性、補助ピン（D/C・RESET）、電源電圧（3.3V）も重要です。

</details>

## まとめ

- 実デバイスとの通信は「CSのLow区間＝1回の取引」で設計し、コマンド＋ダミーのバッファを`transfer`する
- 応答はコマンドの後ろのバイト位置に現れる。転送の`Ok`と「正しい応答」は別物
- モード・周波数・コマンド体系はデータシートが正。3.3V品を使う
