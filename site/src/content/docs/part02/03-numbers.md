## このページでできるようになること

- `u8` / `u32` / `i32` / `f32` などの数値型を目的に合わせて選べる
- オーバーフロー（値が型の範囲からあふれること）が何かを説明できる
- `as` で数値型を変換できる

## 先に結論

Rustの数値型は「符号の有無（uかi）」と「ビット幅（8/16/32/64）」の組み合わせで名前が決まります。`u8` は0〜255、`i32` は約±21億、小数は `f32` / `f64` です。型が違う数値どうしは、C++のように自動では混ぜられず、`as` などで明示的に変換します。面倒に感じますが、「知らないうちに桁があふれる・精度が落ちる」事故を防ぐための仕組みです。マイコンはRAMが小さいので、必要十分な小さい型を選ぶ習慣が役に立ちます。

## 身近なたとえ

数値型は「桁数の決まったカウンター」です。自転車の距離計が4桁なら、9999の次は0000に戻ってしまいます。`u8` は「255までしか数えられないカウンター」で、256を入れようとするとあふれます。これが**オーバーフロー**です。

実際のマイコンでは、カウンターと違って「符号付きかどうか」も型で決まり、あふれたときの動作もビルド設定によって「即座に停止（panic）」か「ぐるっと一周」かが変わる、という点が本物との違いです。

## 仕組み

よく使う数値型を表にまとめます。

| 型 | 意味 | 範囲 | 組み込みでの出番 |
|---|---|---|---|
| `u8` | 符号なし8ビット | 0〜255 | LEDの明るさ、通信の1バイト |
| `u16` | 符号なし16ビット | 0〜65535 | ADC（アナログ入力）の生値など |
| `u32` | 符号なし32ビット | 0〜約42.9億 | 回数、ミリ秒などの一般の数 |
| `u64` | 符号なし64ビット | 0〜約1844京 | `Duration::from_millis` の引数 |
| `i32` | 符号あり32ビット | 約-21.4億〜+21.4億 | 温度など負になりうる数 |
| `f32` | 32ビット浮動小数点 | 小数 | 電圧や温度の計算値 |

- `u` は unsigned（符号なし = 0以上のみ）、`i` は integer の符号あり（負も可）です
- 型注釈を省いた整数は `i32`、小数は `f64` に推論されます
- 数値の後ろに型を直接付ける書き方もあります: `40u8`、`3.3f32`
- 大きな数は `1_000_000` のようにアンダースコアで区切ると読みやすくなります

blinkyの `Timer::after(Duration::from_millis(500))` の `500` は、実は `u64` として渡っています。関数側が `u64` を要求しているので、型推論が自動でそう決めてくれていたのです。

## Arduinoではどう書くか

Arduinoの `int` は多くのボードで16ビットまたは32ビットと、**環境によって幅が変わります**。また型の違う数値を混ぜても自動変換（暗黙の型変換）されるため、気づかないうちに桁あふれや精度落ちが起きることがありました。

```cpp
int brightness = 200;         // intの幅はボード依存
unsigned long ms = 500;       // 「longって何ビット?」も環境依存
```

Rustの型名は `u8` `u32` のように**幅が名前に入っている**ので、どの環境でも同じ意味です。そして型の違う計算はコンパイルエラーになり、変換は必ず自分で書きます。

## RustとEmbassyではどう書くか

これは抜粋です。貼りつけ先の完全なコードは examples/01-blinky を見てください。

```rust
let brightness: u8 = 200;    // 0〜255で十分な値
let wait_ms: u64 = 500;      // 待ち時間
let temperature: f32 = 25.5; // 小数
let diff: i32 = -40;         // 負になりうる値
log::info!("{} {} {} {}", brightness, wait_ms, temperature, diff);

// 型が違う数どうしはasで変換してから計算する
let base: u32 = 1000;
let total = base + brightness as u32;
log::info!("total = {}", total);

// あふれてもよい場面では専用メソッドを使う
let x: u8 = 250;
let y = x.wrapping_add(10); // 250+10=260 → 一周して4になる
log::info!("y = {}", y);
```

## コードを一行ずつ読む

- `let brightness: u8 = 200;` — 0〜255に収まると分かっている値は `u8` で十分。型が仕様書の役割も果たします
- `let total = base + brightness as u32;` — `brightness as u32` で `u8` の値を `u32` に広げてから足しています。小さい型→大きい型への変換は値が変わらず安全です。逆（`u32 as u8`）は上位の桁が捨てられるので注意して使います
- `x.wrapping_add(10)` — 「あふれたら一周する」と明示した足し算。意図があることをコードで表明できます

## よくある失敗

**失敗1: 型の違う数を直接計算した（E0308 / E0277）**

```rust
let base: u32 = 1000;
let brightness: u8 = 200;
let total = base + brightness; // u32 + u8
```

```text
error[E0308]: mismatched types
  |
4 |     let total = base + brightness;
  |                        ^^^^^^^^^^ expected `u32`, found `u8`

error[E0277]: cannot add `u8` to `u32`
  |
4 |     let total = base + brightness;
  |                      ^ no implementation for `u32 + u8`
```

「`u32` を期待したのに `u8` が来た」「`u32 + u8` という足し算は定義されていない」という意味です。C++なら黙って変換して通る場面ですが、Rustは「どちらの型で計算するつもりなのか」を人間に決めさせます。`brightness as u32` と書けば解決です。

**失敗2: 型の範囲を超える値を入れた**

```rust
let x: u8 = 300; // u8は255まで
```

```text
error: literal out of range for `u8`
  |
2 |     let x: u8 = 300;
  |                 ^^^
  |
  = note: the literal `300` does not fit into the type `u8` whose range is `0..=255`
```

`note` が範囲（0..=255）まで教えてくれます。また、計算の**途中**であふれた場合、デバッグビルドでは実行時に止まります。

```text
panicked at src/main.rs:4:13:
attempt to add with overflow
```

これは**panic（パニック）**という「異常を検出して停止する」動作です。あふれる可能性がある計算には、先ほどの `wrapping_add` や、あふれを検出できる `checked_add`（第3部のOptionで扱います）を使います。

## やってみよう

`let ms_per_day = 24 * 60 * 60 * 1000;` を宣言して1日のミリ秒数を表示してみましょう。次に `let x: u8 = 200; let y: u8 = 100;` を足すとどうなるか、コンパイル結果や実行結果を観察してみてください（デバッグビルドでは `attempt to add with overflow` になります）。

## 確認問題

1. `u8` と `i8` の違いは何でしょうか?
2. 0〜4095の値を返すセンサの生値を入れる変数には、表の中のどの型が適切でしょうか?
3. `u32` の値を `u8` に `as` で変換するとき、何に注意が必要でしょうか?

<details>
<summary>答え</summary>

1. `u8` は符号なし（0〜255）、`i8` は符号あり（-128〜127）。同じ8ビットでも表せる範囲が違います。
2. `u16`（0〜65535）。`u8` では足りず、`u32` でも動きますがRAMを余分に使います。
3. 255を超える値は上位の桁が捨てられて別の値になること。小→大は安全、大→小は要注意です。

</details>

## まとめ

- 型名は「符号（u/i/f）+ ビット幅」。範囲が名前から分かる
- 型の違う数値は自動では混ざらない。`as` で明示的に変換する
- あふれ（オーバーフロー）はデバッグビルドでpanicになる。意図があるなら `wrapping_add` 等を使う
