## 先に結論

UDP（User Datagram Protocol）は、IPの小包配送に**ポート番号を付けただけ**の薄いプロトコルです。接続の手続きも、届いたかの確認も、再送も、並べ直しもしません。その代わり、手続きゼロで今すぐ送れて、1通ずつの区切り（データグラム）が保たれ、処理もメモリも軽い。センサ値の定期送信のように「最新の1通が届けばよく、古い1通の紛失は気にしない」用途で輝きます。実は次ページのDNSも、裏ではUDPを使うのが基本です。

## 身近なたとえ

TCPが「電話」なら、UDPは「はがき」です。書いたら出すだけ。相手が読んだかは分かりませんし、2枚出したら順番が入れ替わるかもしれません。でも、電話のように「もしもし」から始める手間はゼロです。

ただし実際のUDPは、はがきと違って**宛先に加えてポート番号で窓口まで指定でき**、1通に載せられるデータもずっと大きい、という点が違います。「IPのはがきに窓口番号を書き足したもの」が正確なイメージです。

## 仕組み

### TCPとUDPの比較

| 観点 | TCP | UDP |
|---|---|---|
| 接続の手続き | あり（3ウェイハンドシェイク） | なし。いきなり送る |
| 届く保証 | あり（確認・再送） | **なし**。失われても分からない |
| 順序 | 保証される | 保証されない |
| データの形 | 区切りのないバイトの流れ | **1通ずつのデータグラム**（区切りが保たれる） |
| 速さ・軽さ | 事務作業のぶん重い | 軽い。遅延が小さい |
| 向く用途 | Webページ、ファイル、確実に届けたい命令 | センサ値の定期送信、探索・呼びかけ、時刻合わせ（NTP）、DNS |

### 「確認しない」が利点になる理由

温度センサの値を1秒ごとに送る場面を考えます。10時0分3秒の値が失われたとして、再送してもらう価値はあるでしょうか。届く頃には10時0分4秒の新しい値が出ています。**古いデータの再送より、次の新しいデータのほうが価値が高い**——こういう用途では、TCPの丁寧さはむしろ遅延のもとです。第9部で学んだ`Signal`（最新値だけ保持）とChannelの使い分けと、同じ発想がネットワークにも現れているわけです。

逆に「設定変更の命令」のような、1通も失えないデータをUDPで送るなら、確認と再送を**自分のアプリで**作り込む必要があります。それはつまりTCPの再発明なので、素直にTCPを使うべき場面です。

## RustとEmbassyではどう書くか（概要）

embassy-netの`udp::UdpSocket`を使います。TCPと似ていますが、2つ違いがあります。①接続という概念がないので、`connect`の代わりに自分の窓口番号を決める`bind`をする。②送受信のたびに相手のアドレスを指定する／受け取る。次の断片はAPIドキュメントに基づく未検証の例です。

```rust
use embassy_net::udp::{PacketMetadata, UdpSocket};

// データ用バッファに加えて、「1通ずつの区切り」を管理する
// メタデータ用バッファも用意する
let mut rx_meta = [PacketMetadata::EMPTY; 8];
let mut rx_buffer = [0u8; 1024];
let mut tx_meta = [PacketMetadata::EMPTY; 8];
let mut tx_buffer = [0u8; 1024];

let mut socket = UdpSocket::new(
    stack,
    &mut rx_meta, &mut rx_buffer,
    &mut tx_meta, &mut tx_buffer,
);
socket.bind(9400)?; // 自分の窓口番号を決める

// 送信: 1通のデータグラムとして届く（相手を毎回指定する）
socket.send_to(b"hello", (remote_addr, 9400)).await?;

// 受信: 何バイト届いたかと、誰から来たかが分かる
let (n, from) = socket.recv_from(&mut buf).await?;
```

TCPにはなかった`PacketMetadata`の配列は、「どこからどこまでが1通か」「誰から来たか」を覚えておくためのメモ帳です。TCPは区切りのない流れなのでデータ用バッファだけで足りましたが、UDPは1通ずつの区切りを保つ必要があるため、メモ帳が別に要ります。

## よくある失敗

- **UDPで送ったのに届かないことがある、をバグだと思う**: 仕様です。UDPは紛失を検知も再送もしません。「失われては困るデータか」をまず考え、困るならTCPにするか、アプリ側で確認の仕組みを作ります
- **1回の`send_to`で送った1通を、相手が2回に分けて受け取ると考える**: TCPと逆の誤解です。UDPはデータグラムの区切りが保たれるので、1通は1通のまま届きます（届けば、ですが）。逆に大きすぎる1通は途中の機器で捨てられやすいので、1通を小さく保つのが定石です
- **`bind`を忘れて受信できない**: 窓口番号を決めないと、相手はどこ宛に送ればいいか分かりません。受信する側は必ず`bind`が必要です

## やってみよう

手元の用途で考えてみましょう。次の3つは、TCPとUDPのどちらで送るべきでしょうか。①1秒ごとの室温、②「ドアを施錠せよ」という命令、③同じネットワーク内の機器を探す「誰かいますか？」の呼びかけ。理由も言えたら合格です（答え: ①UDP ②TCP ③UDP——呼びかけは失敗したらまた呼べばよいためです）。

## 確認問題

1. UDPがTCPより速く・軽くなれるのは、何をしていないからですか。3つ挙げてください。
2. 温度センサの毎秒送信にUDPが向くのはなぜですか。
3. TCPのバッファは2本でしたが、UDPソケットには`PacketMetadata`の配列が追加で必要です。なぜですか。

<details>
<summary>答え</summary>

1. 接続の手続き（ハンドシェイク）、届いたかの確認と再送、順序の並べ直しです。
2. 古い値の再送より次の新しい値のほうが価値が高く、1通の紛失を許容できるからです。再送を待つ遅延がないぶん、軽く送り続けられます。
3. UDPは「1通ずつ」の区切りと送信元の情報を保つ必要があるからです。区切りのないバイトの流れであるTCPと違い、どこまでが1通で誰から来たかを覚えるメモ帳が別に要ります。

</details>

## まとめ

- UDPは「IP＋ポート番号」だけの薄いプロトコル。接続・確認・再送・並べ直しをしない代わりに軽くて速い
- 「最新の1通が届けばいい」データはUDP、「1通も失えない」データはTCP
- embassy-netでは`UdpSocket`を`bind`して`send_to`/`recv_from`で使う（本教材ではコード未検証。概要のみ）
