## このページでできるようになること

- チャタリングが起きる物理的な原因を説明できる
- 「時間によるデバウンス」の考え方を説明できる
- Embassyの`Timer`を使ったデバウンスを実装できる

## 先に結論

機械式ボタンの中では金属の接点同士がぶつかり合っており、押した瞬間・離した瞬間に接点が細かく**バウンド**します。そのため信号は一度でパチッと切り替わらず、数ミリ秒のあいだHigh/Lowが高速に暴れます。これが**チャタリング**です。プログラムがエッジを数えると、1回の押下が何回にも見えてしまいます。対策は「最初のエッジを検出したら、しばらく（この教材では30ms）待ってから状態を確認する」という**時間によるデバウンス**です。Embassyなら`Timer::after`で待つだけで、他のtaskを止めずに実装できます。

## 身近なたとえ

トランポリンに飛び降りたときを想像してください。一度で静止せず、ポン、ポン、と何回か跳ねてから落ち着きます。ボタンの金属接点も同じで、閉じた瞬間に目に見えない速さで数回跳ね、そのたびに電気的には「ついた／離れた」を繰り返します。

たとえと違うのは時間の規模です。トランポリンは数秒跳ねますが、接点のバウンドは**数ミリ秒以下**で収まります。人間には一瞬でも、1秒間に何千万回も命令を実行するマイコンにとっては「何度も切り替わった」とはっきり見える長さです。

## 仕組み

ボタンを1回押したときの信号を拡大すると、こうなっています。

```text
        押した瞬間                       離した瞬間
High ────┐ ┌┐ ┌┐                      ┌┐ ┌┐ ┌──────
         │ ││ ││                      ││ ││ │
Low      └─┘└─┘└──────────────────────┘└─┘└─┘
         ←数ms→                       ←数ms→
        チャタリング     安定してLow    チャタリング
```

エッジを素直に数えるプログラムは、この暴れの一つひとつを「押された！」と誤検出します。1回押しただけでLEDが2回、3回と切り替わる（＝結果的に元に戻ったように見える）のはこのせいです。

**時間によるデバウンス**の考え方は単純です。

1. 最初のフォーリングエッジを検出する
2. 暴れが収まるのに十分な時間（30ms）だけ待つ
3. 待った後で`is_low()`を確認する。まだ押されていれば本物の押下、Highに戻っていればノイズだったと判断する
4. 離すときも同様に、ライジングエッジの後に少し待つ

待ち時間は「チャタリングより長く、人間の連打より短く」選びます。一般的なタクトスイッチの暴れは数ms程度なので、20〜50msあたりがよく使われます。この教材は30msにしています。

## RustとEmbassyではどう書くか

`examples/02-button`のループ部分がそのままデバウンスの実装です。これは抜粋です。完全なコードは `examples/02-button` を見てください。

```rust
loop {
    // High→Lowの変化（＝ボタンが押された瞬間）をawaitで待ちます。
    button.wait_for_falling_edge().await;

    // チャタリング対策: 機械式ボタンは押した瞬間に接点が細かくバタつくので、
    // 30ms待ってから本当に押されているかを確認します。
    Timer::after(Duration::from_millis(30)).await;
    if button.is_low() {
        count += 1;
        led.toggle();
        info!("ボタンが押されました（{}回目）", count);

        // ボタンが離される（Low→High）まで待ってから次の押下を受け付けます。
        button.wait_for_rising_edge().await;
        // 離すときにもチャタリングが起きるので、少し待って落ち着かせます。
        Timer::after(Duration::from_millis(30)).await;
    }
}
```

## コードを一行ずつ読む

```rust
Timer::after(Duration::from_millis(30)).await;
if button.is_low() {
```

デバウンスの本体はこの2行だけです。ポイントは、この30msの待ちが`.await`だということです。C言語やArduinoの`delay(30)`はCPUを止めて待ちますが、Embassyの`Timer::after`は**このtaskだけ**を眠らせます。ボタン処理が待っている30msのあいだも、LED点滅や通信など他のtaskは普通に動き続けます（taskの詳しい話は第9部で扱います）。

```rust
button.wait_for_rising_edge().await;
Timer::after(Duration::from_millis(30)).await;
```

離すときのチャタリング対策です。離した直後の暴れの中のフォーリングエッジを「次の押下」と誤検出しないように、離しを確認してから30ms落ち着かせています。

## 実行方法

```bash
cargo run --release
```

BOOTボタンを何度か押して、**1回の押下でログがちょうど1行**増えることを確認してください。素早く連打しても、押した回数どおりに数えられるはずです。

## よくある失敗

- **デバウンスを外したら1回の押下で複数回反応した**: それが素のチャタリングです。`Timer::after`と`if button.is_low()`を削って試すと体験できます。誤動作ではなく、機械式スイッチの物理現象です
- **待ち時間を長くしすぎて連打を取りこぼす**: たとえば300msにすると、その間の次の押下を検出できません。デバウンス時間は「チャタリングより長く、操作の間隔より短く」が原則です
- **`is_low()`の確認を省略する**: 30ms待つだけで確認をしないと、一瞬のノイズ（電気的な飛び込みなど）でも1回の押下として数えてしまいます。「待ってから確認」までがワンセットです

## やってみよう

`from_millis(30)`を`from_millis(1)`に変えて連打してみてください。カウントが押した回数より増えることがあります（ボタンの個体差で再現しない場合は0msにするか、デバウンス自体を外してみましょう）。確認できたら30msに戻します。

## 確認問題

1. チャタリングの物理的な原因は何ですか。
2. デバウンスの待ち時間を決めるときの原則は何ですか。
3. Arduinoの`delay(30)`によるデバウンスと、Embassyの`Timer::after(...).await`によるデバウンスの最大の違いは何ですか。

<details>
<summary>答え</summary>

1. 機械式スイッチの金属接点が、閉じる（開く）瞬間に細かくバウンドし、数ミリ秒のあいだ接触と非接触を高速に繰り返すことです。
2. 「チャタリングが収まるより長く、想定される操作の間隔より短く」です。短すぎると暴れを拾い、長すぎると次の操作を取りこぼします。
3. `delay(30)`はCPU全体を止めて待つのに対し、`Timer::after(...).await`は待っているtaskだけを眠らせ、その間CPUは他のtaskを実行できることです。

</details>

## まとめ

- チャタリング＝接点のバウンドによる数msの信号の暴れ。1押下が複数回に見える
- 対策は「エッジ検出→30ms待つ→`is_low()`で確認」の時間によるデバウンス
- `.await`で待つので、デバウンス中も他のtaskは止まらない
