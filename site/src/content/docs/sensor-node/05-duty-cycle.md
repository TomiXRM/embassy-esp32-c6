## このページでできるようになること

- デューティサイクル（起きている割合）から平均電流を概算し、電池寿命の桁を見積もれる
- 「時刻が既にあればWi-Fiを一切使わない」という参照元の設計判断を、電流の数字で説明できる
- Deep-sleep前にWi-Fiを止めなければならない理由（止めないと無限ブロック）を説明できる
- 「丸め起床」（09:46に眠って09:50ちょうどに起きる）アルゴリズムの目的と計算を説明できる

## 先に結論

電池駆動の設計で効くのは、瞬間の電流ではなく**平均電流**です。そして平均電流は、回路の細かな工夫より先に、**起きている時間の割合（デューティ比）**でほぼ決まります。データシートの典型値で概算すると、CPUを起こしたままなら約27mA、「30秒ごとに3秒だけ起きる」なら約2.7mA、参照元と同じ「5分ごとに3秒」なら約0.28mA——**眠る割合を変えるだけで、桁が2つ変わります**。こうなると次に効くのは、たまに起きたときに使う大電流の無線です。esp32c3-embassyはここで2つの明確な判断をしています。（1）**時刻がRTC RAMに残っていればWi-Fiを一切使わない**（起動の大半で無線ゼロ）。（2）Wi-Fiを使った回は、**Deep-sleepの前に必ずWi-Fiを止める**（参照元のコメントいわく、止めないと無限にブロックする）。最後に、起床時刻をきりのよい時刻へそろえる「丸め起床」も読みます。

なお、このページの電流値はすべて**ESP32-C6データシートに記載の典型値を使った机上の概算**です。私たちは実測していませんし、開発ボード（USB-UARTブリッジや電源LEDが常時消費）ではこの値になりません（[第12部4ページ](/embassy-esp32-c6/part12/04-power-measurement/)参照）。

## 身近なたとえ

冷蔵庫を考えてください。冷蔵庫の電気代を決めるのは、コンプレッサーの瞬間の消費電力ではなく、「1日のうち何分動いているか」です。庫内が十分冷えていれば止まり、必要なときだけ動く——だから大きな電力の機械なのに電気代は意外と小さい。センサ端末のデューティサイクルはこれと同じ発想です。

たとえと違うのは、マイコンの「止まる」は冷蔵庫より徹底していることです。Deep-sleepではCPUの電源ごと切るので、動作時の**数千分の1**の電流になります。差が極端だからこそ、「動いている割合」がほぼすべてを決めます。

## 平均電流を概算する

[第12部1ページ](/embassy-esp32-c6/part12/01-light-sleep/)の表から、ESP32-C6データシート（v1.5、Table 5-7〜5-11）の典型値を使います。

- 起きている間（CPU 160MHz動作・無線なし＝Modem-sleep相当）: **27mA**
- Deep-sleep（RTCタイマー+LPメモリ有効）: **7µA** = 0.007mA
- Wi-Fi受信時: **78〜82mA**、送信時: **252〜354mA**

平均電流の式は加重平均です。

```text
平均電流 = (起床中の電流 × 起床時間 + スリープ電流 × スリープ時間) ÷ 周期
```

examples/16の設定（3秒起きて30秒眠る…正確には周期33秒ですが、簡単のため30秒周期・起床3秒で計算します）と、参照元の設定（5分周期）を比べます。

| 動かし方 | 計算 | 平均電流（概算） |
|---|---|---|
| 眠らず起きたまま | 27mA × 100% | 約27mA |
| 30秒周期・3秒起床 | (27×3 + 0.007×27) ÷ 30 | **約2.7mA**（1/10） |
| 5分周期・3秒起床 | (27×3 + 0.007×297) ÷ 300 | **約0.28mA**（1/100） |

仮に容量2000mAhの電池1本分のエネルギーを全部使えるとすると（実際は使えません。電圧変換の損失や電池の自己放電を無視した理論値です）、持ち時間は 2000mAh ÷ 平均電流 で、順に**約3日 → 約1か月 → 約10か月**。回路は何も変えていません。**眠る割合を変えただけで桁が動く**——これがデューティサイクル設計の核心で、参照元が「測定は5分に1回で十分」という要件をそのまま「5分眠る」に写し取っている理由です。

もう1つ、この概算から読み取れることがあります。5分周期まで眠ると、内訳は起床分0.27mA＋スリープ分0.007mAで、**まだ起床中の消費が支配的**です。つまり次に削るべきは「起きている3秒の中身」。ここで無線の出番（と問題）が来ます。

## 判断1: 時刻が既にあればWi-Fiを使わない

このプロジェクトでWi-Fiの用途は時刻合わせだけでした（測定データはアップロードしません）。もし毎回の起動でWi-Fi接続をやり直すと何が起きるでしょう。Wi-Fi接続は、アクセスポイントとの接続とDHCPでのアドレス取得（[第10部](/embassy-esp32-c6/part10/02-station/)）だけで数秒かかることが珍しくありません。仮に受信主体で80mAが10秒続くとすると、1周期あたり800mA秒。測定3秒分（27×3=81mA秒）の**約10倍**が時刻合わせだけに消え、せっかくの0.28mAが数mAに逆戻りします。

そこで参照元は、[前ページ](/embassy-esp32-c6/sensor-node/04-rtc-ram/)のRTC RAMを時刻にも使い、**起動時に時刻が残っていればWi-Fiのコードパスを丸ごとスキップ**します（esp32c3-embassy (Claudio Mattera, MIT OR Apache-2.0) src/main.rs より抜粋）。

```rust
let clock = if let Some(clock) = Clock::from_rtc_memory() {
    info!("Clock loaded from RTC memory");
    clock
} else {
    info!("Connect to WiFi");
    let stack = connect_to_wifi(spawner, rng, wifi, (ssid, password)).await?;

    info!("Synchronize clock from server");
    let mut http_client = HttpClient::new(stack, RngWrapper::from(rng));
    let clock = Clock::from_server(&mut http_client).await?;

    info!("Request to disconnect wifi");
    STOP_WIFI_SIGNAL.signal(());

    clock
};
```

電源投入後の最初の1回だけWi-Fiで時刻を取り、以降の起動は**無線ゼロ**。「1ページ目のデューティサイクル図の分岐」の正体はこの`if let`でした。時刻をどうやってRTC RAMに保存し、眠っている時間の分を進めておくのか（`Clock::from_rtc_memory()`の中身）は、後半の[6ページ](/embassy-esp32-c6/sensor-node/06-clock/)の主役です。

## 判断2: 眠る前にWi-Fiを必ず止める

上の抜粋の`STOP_WIFI_SIGNAL.signal(())`に注目してください。時刻が取れたら、Wi-Fi接続を維持しているtaskへ「もう止めてよい」と合図を送っています。合図を受けたtask側は`controller.stop_async().await`でWi-Fiを停止します（esp32c3-embassy src/wifi.rs）。そして眠りに入る関数には、こんな注意書きがあります（esp32c3-embassy (Claudio Mattera, MIT OR Apache-2.0) src/sleep.rs より）。

```rust
/// Enter deep sleep for the specified interval
///
/// **NOTE**: WiFi must be turned off before entering deep sleep, otherwise
/// it will block indefinitely.
pub fn enter_deep(rtc_cntl: LPWR, interval: Duration) -> ! {
```

「Wi-Fiを止めずにDeep-sleepに入ろうとすると、無限にブロックする」。これは作者が実際に踏んだ落とし穴の記録でしょう。無線スタックが動いたままではチップは安全に電源を落とせません。**起動（電源を入れる）と終了（後片付けして眠る）は対称**で、無線のような大きなサブシステムほど「使い終わったら止めてから眠る」を明示的にやる必要があります。Signalで停止を依頼し、taskが停止を完了してから眠る——[第9部](/embassy-esp32-c6/part09/09-channel-signal-mutex/)で学んだ道具が、省電力の文脈で再登場しています。

## 丸め起床 — きりのよい時刻に起きる

最後に、参照元のもう1つの工夫を紹介します。単純に「300秒眠って起きる」を繰り返すと、起床時刻は09:46:12、09:51:12、09:56:12…と**中途半端な時刻がずっと続きます**。測定記録としては、09:50:00、09:55:00のような「きりのよい時刻」にそろっているほうが、グラフにしたときも、他の機器の記録と突き合わせるときも、欠測を見つけるときも扱いやすい。そこで参照元は「次のきりのよい時刻までの残り時間」を計算します（esp32c3-embassy (Claudio Mattera, MIT OR Apache-2.0) src/clock.rs より）。

```rust
/// Compute the next wakeup rounded down to a period
///
/// * At 09:46:12 with period 1 minute, next rounded wakeup is 09:47:00.
/// * At 09:46:12 with period 5 minutes, next rounded wakeup is 09:50:00.
/// * At 09:46:12 with period 1 hour, next rounded wakeup is 10:00:00.
fn next_rounded_wakeup(now: Duration, period: Duration) -> Duration {
    let then = now + period;
    Duration::from_secs((then.as_secs() / period.as_secs()) * period.as_secs())
}
```

計算は2手です。（1）今の時刻（Unixエポック秒）に周期を足す。（2）その値を周期で**切り捨て割り**して周期の倍数に丸める（整数の割り算`/`は余りを捨てるので、`(x / p) * p`は「pの倍数へ切り捨て」になります）。09:46:12に5分（300秒）を足すと09:51:12、これを300秒の倍数に切り捨てると09:50:00。「今+周期を、周期の目盛りへ切り捨てる」だけで、必ず「次の目盛りの時刻」が得られます。参照元は起きている間の測定タイミングにこの計算を使っており、毎回「次の目盛りまでの残り」を計算し直すので、**ずれが蓄積しない**という利点もあります。時刻という土台（RTC RAMに保存した壁時計）があって初めて成立する工夫です。

## よくある失敗

1. **瞬間電流だけ見て電池を選ぶ** — Wi-Fi送信の瞬間は250mA超でも、それが周期のごく一部なら平均への寄与は小さいことがあります。逆に、数mAの「常時流れる小さな漏れ」（例: 開発ボードの電源LED）は平均をそのまま押し上げます。効くのは常に「電流×時間の合計」です
2. **概算値を実測値と思い込む** — このページの数字はチップ単体のデータシート典型値による机上計算です。実際の平均電流は、ボード上の他部品、電源回路の効率、無線環境（接続にかかる時間）で大きく変わります。設計の当たりを付けるのが概算、確かめるのが実測（第12部4ページ）です
3. **Wi-Fiを止めずにDeep-sleepへ入ろうとする** — 参照元のコメントのとおり、無限ブロックの原因になります。「spawnしたtaskに止まってもらってから眠る」という終了手順を設計に含めてください
4. **毎回Wi-Fiでデータ送信する設計にして電池が持たない** — 送信が本当に毎回必要かを疑うのが先です。RTC RAMに数回分ためて、N回に1回まとめて送るだけでも、無線の起動回数は1/Nになります

## やってみよう

自分の「概算表」を作ってみましょう。examples/16の`SLEEP_TIME`を60秒・300秒に変えた場合と、`AWAKE_TIME`を3秒から1秒に削った場合について、このページの式で平均電流を計算してください（データシート典型値27mA/7µAを使用）。「眠りを延ばす」と「起床を削る」のどちらが今効くのか、内訳から判断できるはずです。

## 確認問題

1. 「5分周期・3秒起床」の概算で、平均電流の内訳の大部分を占めるのはスリープ分・起床分のどちらですか。それは次に何を削るべきだと示していますか。
2. 参照元が2回目以降の起動でWi-Fiを使わずに済むのは、何をどこに保存しているからですか。
3. 丸め起床の計算「(now + period) を period で切り捨て」で、09:58:30・周期5分のときの起床時刻を求めてください。

<details>
<summary>答え</summary>

1. 起床分です（約0.27mA。スリープ分は約0.007mA）。次に削るべきは「起きている時間の長さと中身」であり、特に無線のような大電流の処理を起床中に入れない工夫が効きます。
2. 時刻（起動時刻のエポック秒とタイムゾーンオフセット）をRTC RAMに保存しているからです。時刻が残っていれば、Wi-Fiの唯一の用途（時刻合わせ）が不要になります。
3. 09:58:30 + 5分 = 10:03:30。これを5分（300秒）の目盛りへ切り捨てると10:00:00です。

</details>

## まとめ

- 平均電流は「電流×時間の加重平均」。デューティ比を下げるだけで27mA→2.7mA→0.28mAと桁で下がる（すべてデータシート典型値による概算で、実測ではない）
- 参照元の2大判断: 時刻がRTC RAMにあればWi-Fiを一切使わない（起動の大半で無線ゼロ）、Wi-Fiを使ったら止めてから眠る（止めないと無限ブロック）
- 丸め起床は「今+周期を周期の目盛りに切り捨て」。記録がそろい、ずれも蓄積しない。壁時計を持つ端末ならではの工夫
