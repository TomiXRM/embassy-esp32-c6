## このページでできるようになること

- `Instant::now()`が「壁の時計」ではない理由と、それでも時刻が計算できる仕組みを説明できる
- ネットワークから一度だけ時刻を取得し、以後は自力で時刻を維持する`boot_time`方式を説明できる
- ディープスリープをまたいで時刻を持ち越すために、参照元が「保存する値にひと工夫」した理由を説明できる
- 測定時刻をきれいな時刻（毎時0分・5分…）に揃える「丸め起床」の計算ができる

## 先に結論

ESP32-C6には、電池でずっと動き続けるカレンダー時計は載っていません。第9部6ページで学んだとおり、`Instant::now()`が返すのは**起動した瞬間からの経過時間**だけです。参照元のesp32c3-embassyは、初回起動時にWi-Fi経由のHTTPSで現在時刻（Unixエポック秒）を一度だけ取得し、`boot_time = 現在時刻 − Instant::now()`を計算して覚えます。以後は`now() = boot_time + Instant::now()`の足し算だけで、通信なしに現在時刻が分かります。さらにディープスリープ前にこの時刻を**RTC RAM**へ保存するので、次回以降の起動ではWi-Fiに一切つながずに時計を引き継げます。保存するのは「今の時刻」ではなく「今＋予定スリープ時間」——この一工夫がこの設計のいちばん面白いところです。

## 身近なたとえ

手元にあるのがストップウォッチだけだと想像してください。ストップウォッチは「押してから何秒たったか」しか教えてくれません。でも、通りがかりの人に一度だけ「いま何時ですか？」と聞いてメモしておけば、あとは「聞いた時刻＋ストップウォッチの経過」でいつでも現在時刻が分かります。これが`boot_time`方式です。

たとえと違うのは、マイコンは**眠る（ディープスリープする）とストップウォッチまでリセットされる**ことです。人間なら寝ても腕時計は動き続けますが、C6ではスリープ復帰後に`Instant`が0から再スタートします。だから参照元は、寝る前に「起きる予定の時刻」をメモ（RTC RAM）に書いてから眠ります。

## 仕組み

### Instantは壁時計ではない — 第9部の回収

`embassy_time::Instant`は単調増加のカウンタで、「2026年7月18日 9時46分」のような**壁時計（カレンダー時刻）**とは別物です。ログのタイムスタンプや周期処理には十分ですが、「毎時0分に測定したい」「測定値に日時を付けてクラウドへ送りたい」となった瞬間に足りなくなります。C6のRTC（Real-Time Clock）ドメインにはタイマーはあっても、電源を切っても進み続ける時計はありません。時刻は**外からもらう**しかないのです。

### 一度だけ聞いて、あとは足し算 — Clockの中心

参照元の`clock.rs`の中心部はこれだけです（出典: esp32c3-embassy `src/clock.rs`、Claudio Mattera、MIT OR Apache-2.0。以下の引用も同じ）。

```rust
// これは抜粋です（clock.rs）
pub fn new(current_time: u64, offset: UtcOffset) -> Self {
    let from_boot = Instant::now().as_secs();
    let boot_time = current_time - from_boot;
    Self { boot_time, offset }
}

pub fn now_as_epoch(&self) -> u64 {
    let from_boot = Instant::now().as_secs();
    self.boot_time + from_boot
}
```

`current_time`はサーバからもらったUnixエポック秒（1970年1月1日0時（UTC）からの通算秒）です。そこから「起動してから今までの秒数」を引くと、**起動した瞬間が何時だったか**（`boot_time`）が求まります。以後の`now()`は`boot_time + 経過秒`の足し算だけ。通信は初回の一度きりです。

時刻の取得は`Clock::from_server()`がHTTPSクライアント経由で行います（HTTPSそのものは次ページで扱います）。取得後はWi-Fiを止めてよいので、電池の節約にも直結します。

### ディープスリープを生き残る — 保存値のひと工夫

前ページまでで見たとおり、ディープスリープに入るとメインメモリは消え、復帰後はプログラムが最初から走り直します。`boot_time`もろとも消えるので、そのままでは毎回Wi-Fi接続が必要になってしまいます。参照元はスリープ直前に`#[ram(unstable(rtc_fast))]`を付けたRTC RAM上のstaticへ時刻を保存します。

```rust
// これは抜粋です（clock.rs）
pub fn save_to_rtc_memory(&self, expected_sleep_duration: Duration) {
    let now = self.now_as_epoch();
    let then = now + expected_sleep_duration.as_secs();
    // …thenをRTC RAMのBOOT_TIMEへ書き込む…
}
```

保存しているのは`now`ではなく`then = now + 予定スリープ時間`、つまり**次に目覚める予定の時刻**です。なぜでしょうか。復帰後の起動では`Instant`が0から再スタートします。目覚めた瞬間は「起動0秒目」なので、そのときの現在時刻＝`boot_time`そのもの。予定どおりに目覚めたなら、RTC RAMに入っている「目覚める予定の時刻」をそのまま新しい`boot_time`として`Clock::new()`に渡せる、という寸法です。復帰側の`from_rtc_memory()`は、値が0なら「まだ一度も同期していない」と判断して`None`を返し、初回だけWi-Fi同期に回します。

正直に書くと、この方式の時刻は少しずつずれます。ディープスリープ中の計時はRTC用の低速クロックに頼っており、その周波数には誤差があるからです。数分〜数時間の周期で表示に使うぶんには十分ですが、長期間動かすなら「ずれが積もったら再同期する」設計（例: 起動N回ごとにWi-Fi同期をやり直す）を足すことになります。

### 丸め起床 — きれいな時刻に目覚める

測定周期が5分でも、起動した時刻が09:46:12なら、そのまま5分おきに測ると09:51:12、09:56:12…と中途半端な時刻が並びます。参照元は「次のキリのよい時刻までの残り時間」を計算して、そのぶんだけ待ちます（`clock.rs`の107〜138行目）。

```rust
// これは抜粋です（clock.rs）
// 09:46:12・周期5分 → 次の丸め起床は09:50:00
fn next_rounded_wakeup(now: Duration, period: Duration) -> Duration {
    let then = now + period;
    Duration::from_secs((then.as_secs() / period.as_secs()) * period.as_secs())
}
```

整数の割り算が小数点以下を切り捨てることを利用しています。「今＋周期」を周期で割って掛け直すと、周期の倍数に切り下げられます。09:46:12（エポック秒で…12秒）に5分（300秒）を足すと09:51:12。300で割って掛けると09:50:00に切り下がる、という動きです。測定時刻が00分・05分・10分…に揃うので、グラフにしたときの見栄えと比較のしやすさが段違いです。

### NTPとの関係 — 本物と簡易版

世の中の機器の時刻合わせの本命は**NTP（Network Time Protocol）**、組み込みでは簡略版の**SNTP**（RFC 4330）です。UDPで時刻サーバと往復し、通信にかかった往復時間まで補正して数十ミリ秒の精度を出します。参照元はこれを使わず、HTTPSのAPI（Adafruit IOの時刻API）から時刻をもらう簡易方式にしています。精度は秒単位に落ちますが、「気象データに付ける時刻」には十分で、どのみち使うHTTPSクライアントを使い回せるぶん実装が小さく済みます。ここでも層の区別を思い出してください。Wi-Fiはリンクをつなぐ層、その上にUDP（SNTP）やTCP＋TLS（HTTPS）が乗り、時刻の受け渡しはさらにその上の話です。

なお、このページの内容はexamplesに対応する検証済みコードがありません（`code_status: concept-only`）。引用はすべて参照元プロジェクトのものです。

## よくある失敗

- **`Instant::now()`をそのまま時刻として保存してしまう** — `Instant`は起動からの経過時間なので、別の起動サイクルの`Instant`とは比較できません。壁時計が必要なら必ず`boot_time`のような基準値と組にします
- **スリープ復帰後に`boot_time`を計算し直さない** — 復帰後は`Instant`が0に戻っています。スリープ前の`boot_time`をそのまま使うと、時刻がスリープぶんだけ過去に飛びます。「予定復帰時刻を保存しておき、復帰時にそれを新しい`boot_time`にする」のが参照元の答えです
- **電源を抜いても時刻が残っている前提で書く** — RTC RAMが生きるのはディープスリープの間だけで、電源を抜けば消えます。参照元が「値が0なら未同期」とチェックしているのはこのためです（エポック秒0＝1970年が正しい現在時刻であることはまずないので、0を「未初期化」の印に使えます）

## やってみよう

紙と電卓で丸め起床を計算してみましょう。いまがエポック秒`1_752_800_777`（10分の区切りから6分17秒過ぎた時刻）で周期が10分（600秒）のとき、`next_rounded_wakeup`は何を返しますか。`(1_752_800_777 + 600) / 600 * 600`を計算し、元の値との差（＝待ち時間）が「次の10分の倍数の時刻まで」になっていることを確かめてください。

## 確認問題

1. `boot_time = サーバ時刻 − Instant::now()`という引き算は、何を求めていますか。
2. `save_to_rtc_memory()`が「今の時刻」ではなく「今＋予定スリープ時間」を保存するのはなぜですか。
3. 参照元がSNTPではなくHTTPSで時刻を取るのは、何を優先した設計判断ですか。失うものは何ですか。

<details>
<summary>答え</summary>

1. 「起動した瞬間の壁時計時刻」です。以後はこれに経過時間を足すだけで現在時刻が求まります。
2. ディープスリープ復帰後は`Instant`が0から再スタートするため、目覚めた瞬間の現在時刻がそのまま新しい`boot_time`になるからです。予定復帰時刻を保存しておけば、復帰時の計算が不要になります。
3. 実装の小ささを優先しました。時刻取得にもデータ送信にも同じHTTPSクライアントを使い回せます。失うのは精度（SNTPの往復遅延補正がなく、秒単位のずれが出る）です。

</details>

## まとめ

- `Instant`は起動からの経過時間。壁時計は「一度だけ外から聞いて、あとは足し算」で作る
- ディープスリープ前に「予定復帰時刻」をRTC RAMへ保存すれば、次回以降はWi-Fiなしで時計を引き継げる
- 丸め起床は整数割り算の切り捨てを使って、測定時刻をきれいな周期に揃えるテクニック
