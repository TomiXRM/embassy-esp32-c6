## このページでできるようになること

- C++の主要な仕組み（class、仮想関数、null、例外、template、RAII、move）とRustの対応物を言える
- 対応するだけでなく「何が違うのか」「Rustの制約はなぜあるのか」を説明できる
- どちらの言語にも得意な場面があることを、具体例で説明できる

## 先に結論

ArduinoのコードはC++です。つまりあなたはすでにC++の入り口に立っていて、第2部から学んできたRustの機能の多くは、C++の仕組みと**きれいに対応**します。このページはその対応表です。ただし「Rustの方が偉い」という話ではありません。両者は多くの目標を共有していて、違うのは**間違いをいつ捕まえるか**（コンパイル時か、実行時か、レビューか）という設計判断です。Rustはコンパイル時に寄せた分だけ書くときの制約が強く、C++は自由な分だけ守るべき約束が人間側に残ります。各項目で、C++の考え方・Rustの考え方・制約の理由・Rustでも残る問題・組み込みでの向き不向きを順に見ます。

## 身近なたとえ

2種類の自転車を想像してください。片方は補助輪や反射材やベルの規格が厳しく、規格を満たさないと出荷できません（Rust）。もう片方は自由に改造できて部品も豊富ですが、安全確認は乗る人の腕にかかっています（C++）。どちらも「速く安全に走る」という目的は同じです。

実際の技術との違いを一言添えると、この2つは優劣ではなく**検査のタイミング**の違いです。またRustの検査はコンパイル時に行われるため、走り（実行速度）を遅くする補助輪ではない、という点も自転車のたとえとは違います。

## 比較の前に

- C++の文法を完全に知らなくても大丈夫です。C++側のコードは雰囲気が伝わる最小限にしてあります。
- Rust側のコードはすべてホストPCでコンパイル確認済みです（Playgroundでも動きます）。C++側は対比のための抜粋です。
- 各項目の深掘りは該当ページへのリンクを貼ります。このページは「地図」です。

## 1. class ↔ struct + impl

```cpp
// C++
class Motor {
  uint8_t speed = 0;
public:
  void set_speed(uint8_t s) { speed = s; }
};
```

```rust
// Rust
struct Motor {
    speed: u8,
}

impl Motor {
    fn set_speed(&mut self, s: u8) {
        self.speed = s;
    }
}
```

1. **C++では**: classがデータとメソッドと公開範囲を1つの箱にまとめます。継承で親classの機能を受け継ぐ設計が中心にあります。
2. **Rustでは**: データ（struct）と振る舞い（impl）を分けて書きます。継承はなく、共通化はtraitで行います（[第3部6ページ](/embassy-esp32-c6/part03/06-methods/)）。
3. **制約の理由**: 継承を外したのは、深い継承階層で「どのメソッドが実際に呼ばれるか」が追いにくくなる問題を、言語ごと避けるためです。
4. **Rustでも解決できない問題**: 「良い設計の分割」は言語が決めてくれません。structの切り方が悪ければRustでも読みにくいコードになります。また継承ベースの既存設計（GUIフレームワーク等）の移植は素直に書けず苦労します。
5. **組み込みでは**: ほぼ引き分けです。継承を深く使う設計はC++の組み込み現場でも避けられる傾向があり、実際の書き味は近くなります。既存のC++資産（ESP-IDF等）に乗るならC++が直接的です。

## 2. 仮想関数・interface ↔ trait

```cpp
// C++
class Drive {
public:
  virtual void stop() = 0; // 純粋仮想関数
};
```

```rust
// Rust
trait Drive {
    fn stop(&mut self);
}
```

1. **C++では**: 純粋仮想関数を持つ抽象classがinterfaceの役目をします。呼び出しは基本的に実行時のvtable経由です。
2. **Rustでは**: traitが約束を定義します（[5ページ](/embassy-esp32-c6/part04/05-trait/)）。既定はジェネリクス（コンパイル時解決）で、実行時解決は `dyn` と明示したときだけです。
3. **制約の理由**: コンパイル時と実行時のどちらで解決するかを書き分けさせるのは、コストが見えない呼び出しを作らないためです。
4. **Rustでも解決できない問題**: traitの設計そのものの難しさは残ります。またC++のテンプレート+仮想関数の柔軟な混合技法の一部は、Rustでは書き直しが必要です（dyn化できないtraitがあるなど）。
5. **組み込みでは**: embedded-halのようにジェネリクス既定のRust方式はフラッシュとRAMの見積もりがしやすい一方、C++の仮想関数もコストは小さく実績豊富です。既存のC++ HAL資産を使う場面ではC++が有利です。

## 3. null ↔ Option

```cpp
// C++
Sensor* find_sensor();      // 見つからなければ nullptr
auto* s = find_sensor();
s->read(); // nullチェックを忘れても書けてしまう
```

```rust
// Rust
fn find_sensor() -> Option<Sensor>;
// 使う側はSomeかNoneかを処理しないと中身に触れない
```

1. **C++では**: 「無い」をnullptrで表し、使う前に人間がチェックします。近年は `std::optional` も使えますが、生ポインタとの混在は続いています。
2. **Rustでは**: 「無いかもしれない」は型 `Option<T>` で表し、チェックせずに中身を使うコードはコンパイルできません（[第3部4ページ](/embassy-esp32-c6/part03/04-option/)）。
3. **制約の理由**: nullチェック忘れは「たまにしか起きないのに致命的」というバグの代表格で、人間の注意力に頼らない仕組みが必要だからです。
4. **Rustでも解決できない問題**: `unwrap()` と書けばチェックを放棄でき、実行時パニックになります。「Noneのときどうするべきか」という仕様の判断も言語は肩代わりしません。
5. **組み込みでは**: デバッガを繋ぎにくい現場ではnull起因のクラッシュの再現・調査が特に高くつくため、コンパイル時に落とせるRust方式の利点が大きい場面です。C++でも `std::optional` を徹底すれば近づけますが、強制はされません。

## 4. 例外 ↔ Result

```cpp
// C++
try {
  init_device(0);
} catch (const ConfigError& e) {
  // どの関数がthrowするかは型からは分からない
}
```

```rust
// Rust
match init_device(0) {
    Ok(freq) => { /* 続行 */ }
    Err(e) => { /* 失敗が型で見える */ }
}
```

1. **C++では**: 失敗はthrowで投げ、どこかのcatchで受けます。正常系のコードから失敗処理を分離できるのが利点です。
2. **Rustでは**: 失敗は戻り値 `Result` です（[第3部5ページ](/embassy-esp32-c6/part03/05-result/)、[8ページ](/embassy-esp32-c6/part04/08-error-design/)）。関数の型を見れば失敗しうるかが分かり、`?` で伝播します。
3. **制約の理由**: 例外は「この関数は失敗するか」が署名から見えず、飛び先も文面から追えません。失敗を型に載せるのは、呼び出し側に処理の判断を強制するためです。
4. **Rustでも解決できない問題**: パニック（配列の範囲外アクセス等）という第2の失敗経路は残っています。またResultの連鎖だけでは表現しにくい「深い場所から一気に抜ける」処理は、例外の方が短く書けることがあります。
5. **組み込みでは**: 実は差が縮む領域です。組み込みC++は例外を無効化（-fno-exceptions）してエラーコード戻り値で書く現場が多く、Resultはそのスタイルを言語が支援した形といえます。チェック忘れをコンパイラが検出する分Rustが楽ですが、C++の慣行と発想は同じです。

## 5. template ↔ generics

```cpp
// C++
template <typename T>
T max_of(T a, T b) { return a > b ? a : b; }
// Tに>が無い場合、エラーは使った時に内部から噴き出す
```

```rust
// Rust
fn max_of<T: PartialOrd>(a: T, b: T) -> T {
    if a > b { a } else { b }
}
// Tの条件は署名に書く。違反は呼び出し行で1つのエラーになる
```

1. **C++では**: templateは「コードの型抜き」で、実際に使われた時点で型を当てはめて検査します。C++20のconceptsで条件を書けるようになりましたが、省略も可能です。
2. **Rustでは**: ジェネリクスは**宣言時に**トレイト境界で条件を明示します（[6ページ](/embassy-esp32-c6/part04/06-generics/)）。境界にない操作は本体に書けません。
3. **制約の理由**: 条件を署名に固定すると、エラーが「呼び出し側の1行」に集約され、ライブラリ内部の長大なエラーが利用者に噴き出さないためです。
4. **Rustでも解決できない問題**: コードサイズの増加（単相化）は同じように起きます。またC++templateの一部の高度な技法（特殊化など）に相当する機能は、Rustでは安定版にありません。
5. **組み込みでは**: 型ごとの複製によるフラッシュ消費という悩みは両者共通です。エラーメッセージの読みやすさではRust、既存のC++テンプレートライブラリ資産の厚みではC++に分があります。

## 6. デストラクタ・RAII ↔ Drop

```cpp
// C++
{
  BusGuard guard; // コンストラクタで確保
} // スコープを抜けるとデストラクタで解放（RAII）
```

```rust
// Rust
{
    let _guard = BusGuard;
} // スコープを抜けると必ずdropが呼ばれる
```

1. **C++では**: RAII（確保は構築時に、解放は破棄時に）はC++が生んだ代表的な設計で、ロックやバスの解放をデストラクタに任せます。
2. **Rustでは**: 同じ発想を `Drop` traitで書きます。所有権があるため「誰がいつdropするか」が常に一意に決まります（[第3部8ページ](/embassy-esp32-c6/part03/08-ownership/)）。
3. **制約の理由**: C++のRAIIは「コピーやポインタ経由で二重解放・解放後使用が起こりうる」という穴があり、所有権の一意性はそれを塞ぐための土台です。
4. **Rustでも解決できない問題**: dropの実行はプログラムの正常な進行が前提です。電源断や無限ループでは走りません。「解放の順序を跨いだ設計ミス」も検出対象外です。
5. **組み込みでは**: ペリフェラルの後始末やロック解放にどちらも有効で、発想は共通です。RustはGPIOピンの「二重使用」を所有権で防げる（esp-halの設計）点が組み込み特有の利点です。

## 7. コピー既定 ↔ move既定

```cpp
// C++
UartPort a;
UartPort b = a; // 既定はコピー。aもbも生きている
// moveしたい時は std::move(a) と明示。ただしmove後のaも触れてしまう
```

```rust
// Rust
let a = UartPort { id: 1 };
let b = a; // 既定はmove。以後aに触るとコンパイルエラー
```

1. **C++では**: 代入の既定はコピーで、moveは `std::move` で明示します。move後の変数も文法上は使えてしまい、使わないのは慣習です。
2. **Rustでは**: 既定がmoveで、moveした後の変数はコンパイラが使用禁止にします（[第3部8ページ](/embassy-esp32-c6/part03/08-ownership/)）。
3. **制約の理由**: 「同じ資源を2か所が持っている」状態を作らないためです。UARTのような実物のペリフェラルは世界に1つしかなく、コピーできる方が不自然だ、という考え方です。
4. **Rustでも解決できない問題**: 借用チェッカが安全なコードを弾くことがあり、設計の書き直し（構造の持ち方の変更）を強いられます。この学習コストと設計コストは実在します。
5. **組み込みでは**: 「ペリフェラルは1つ」という現実とmove既定の相性は良く、esp-halはこの性質を安全性の根拠に使っています。一方、素早い試作でとにかく書き飛ばしたい場面では、制約の少ないC++（Arduino）の方が手が速いのも事実です。

## 全体の対応表

| C++ | Rust | 検査のタイミングの違い |
|---|---|---|
| class | struct + impl | 継承なし。共通化はtraitで |
| 純粋仮想関数 / interface | trait | 既定がコンパイル時解決（dynは明示） |
| nullptr / std::optional | Option | チェックしないと中身に触れない |
| 例外 / エラーコード | Result + ? | 失敗が関数の型に現れる |
| template / concepts | generics + トレイト境界 | 条件を署名に強制 |
| デストラクタ / RAII | Drop + 所有権 | 解放責任者が常に一意 |
| コピー既定 + std::move | move既定 | move後の使用はコンパイルエラー |

## よくある失敗

**1. 「Rustならバグが出ない」と思い込む**

Rustが防ぐのは主にメモリと型の誤用です。仕様の勘違い、計算式の誤り、ハードウェアの理解不足によるバグは、この教材の後半でも普通に登場します。「コンパイルが通った＝正しい」ではありません。

**2. C++の知識を捨てようとする**

RAII、エラーコード戻り値、実行時コストへの意識——組み込みC++の良い習慣は、そのままRustの設計の源流です。Arduino/C++の経験は荷物ではなく土台です。対応表を手に「あの考え方はRustではこう書く」と翻訳していくのが最短の学習路です。

## やってみよう

7項目のうち1つを選び、C++側の「人間が守る約束」をわざと破るとどうなるか（実行時に何が起きうるか）と、Rust側で同じことをするとどのエラーになるかを、ノートに1行ずつ書いてみましょう。3番（null）なら「チェック忘れ→実行時クラッシュ」対「unwrapなしでは書けない→コンパイルエラー」です。

## 確認問題

1. C++の純粋仮想関数に対応するRustの仕組みは何ですか？ 呼び出しの解決タイミングの既定はどう違いますか？
2. 組み込みC++で例外がしばしば無効化されることを踏まえると、Resultはどう位置づけられますか？
3. 「Rustでも解決できない問題」を2つ挙げてください。

<details>
<summary>答え</summary>

1. traitです。C++の仮想関数は実行時解決が基本、Rustはコンパイル時解決（ジェネリクス）が既定で、実行時解決はdynで明示します。
2. 「エラーコードを戻り値で返す」という組み込みC++の慣行を、チェック忘れをコンパイラが検出できる形にしたもの、と位置づけられます。発想は連続しています。
3. 例: 仕様の誤解によるバグ。unwrapによる実行時パニック。設計分割の巧拙。電源断時のdrop不実行。ジェネリクスによるコードサイズ増。既存C++資産との接続コスト。
</details>

## まとめ

- C++とRustの主要機能は対応表が引ける程度によく似ており、目標も共有している
- 違いの本質は「約束を人間が守る（C++）か、コンパイラが強制する（Rust）か」という検査タイミングの設計
- Rustにも解けない問題と学習コストがあり、C++の経験と資産はRust学習の土台になる
