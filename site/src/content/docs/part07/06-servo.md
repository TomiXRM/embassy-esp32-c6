## このページでできるようになること

- サーボが「50Hzのパルス幅」で角度指令を受け取る仕組みを説明できる
- LEDCの設定を50Hz用に変えてサーボを動かす方針が分かる
- サーボの電源をUSB 5Vから取ってはいけない理由を説明できる

**注意: このページのコードは検証済みexampleを50Hz用に書き換えた未検証コードです（構文レビューのみ、cargo check・実機確認なし）。動作を保証しません。**

## 先に結論

ホビーサーボ（SG90など）は、**50Hz（20ms周期）**のPWM信号の**パルス幅**で角度が決まるモーターです。目安はパルス幅1.0msで片端、1.5msで中央、2.0msで反対端です（個体差があります）。LEDと違い、duty比の「平均エネルギー」ではなく「Highの時間そのもの」が命令になります。LEDCのタイマーを50Hzに設定すれば同じ仕組みで駆動できますが、`set_duty`は整数パーセント指定のため角度が粗くしか指定できません。また、サーボのモーター電源は外部5Vから取り、GNDだけをボードと共通にします。

## 身近なたとえ

サーボへの信号は、犬の訓練の笛のようなものです。笛を「短く吹けば右、長く吹けば左」と教えられた犬は、笛の音の**長さ**を聞いて動きます。サーボも20msごとに届くパルスの**幅**（Highの長さ）を測って、指示された角度へ動きます。

ただし犬と違い、サーボは内部に位置センサと制御回路を持っていて、現在の角度と指令の差を自分で修正し続けます。パルスを送り続ける限り、外から押されても指定角度へ戻ろうとします。

## 仕組み

### パルス幅が角度になる

```text
|←──────── 20ms（50Hz）────────→|
▔▔▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁   1.0ms → 片端（目安）
▔▔▔▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁   1.5ms → 中央（目安）
▔▔▔▔▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁   2.0ms → 反対端（目安）
```

- 周波数は**50Hz固定**が標準です。LEDのように周波数を自由に選べません
- 角度を決めるのはduty比ではなく**パルス幅（ms）**です。同じ1.5msでも、周期が20msだからduty 7.5%になる、という関係です
- 実際の可動範囲はパルス幅0.5〜2.4msまで反応する個体もあります。端まで動かすときは、ジジジと音を立てて突き当たっていないか確認してください（突き当たったまま保持すると壊れます）

### LEDCでの実現方針と分解能の壁

やることは[4. PWMとは何か](/embassy-esp32-c6/part07/04-pwm/)と同じで、変更点は2つだけです。

1. タイマーの周波数を5kHzから**50Hz**へ
2. ピンをGPIO10から**GPIO11**へ（教材のピン割り当て）

問題は`set_duty`が**整数パーセント**しか受け取れないことです。20ms周期では1% = 0.2msなので、指定できるパルス幅は5%=1.0ms、6%=1.2ms、…、10%=2.0msの**6段階**しかありません。中央の1.5ms（7.5%）は指定できず、7%（1.4ms）か8%（1.6ms）で代用することになります。「数段階の首振り」はできますが、滑らかな角度制御にはduty値を細かく指定できる低レベルAPIが必要です（この教材の検証範囲外とし、ここでは扱いません）。

## RustとEmbassyではどう書くか

**以下は`examples/13-adc-pwm`の検証済みコードを50Hz・GPIO11用に書き換えた未検証の参考コードです。cargo checkを通していません。**

```rust
// タイマー0を「50Hz・12bit分解能」に設定（未検証）
let mut lstimer0 = ledc.timer::<LowSpeed>(timer::Number::Timer0);
lstimer0
    .configure(timer::config::Config {
        duty: timer::config::Duty::Duty12Bit,
        clock_source: timer::LSClockSource::APBClk,
        frequency: Rate::from_hz(50),
    })
    .unwrap();

// サーボの信号線はGPIO11
let mut channel0 = ledc.channel(channel::Number::Channel0, peripherals.GPIO11);
channel0
    .configure(channel::config::Config {
        timer: &lstimer0,
        duty_pct: 7, // 約1.4ms ≒ 中央付近
        drive_mode: DriveMode::PushPull,
    })
    .unwrap();

loop {
    channel0.set_duty(5).unwrap(); // 約1.0ms → 片端
    Timer::after(Duration::from_millis(1000)).await;
    channel0.set_duty(10).unwrap(); // 約2.0ms → 反対端
    Timer::after(Duration::from_millis(1000)).await;
}
```

構造は検証済みのLED調光コードと同一で、数値だけが違います。試す場合は、まずサーボを外して書き込み、動作ログを確認してからサーボをつなぐと安全です。

## 配線

**サーボの電源はボードの5Vピンから取らないでください。**サーボのモーターは動き出しの瞬間に大きな電流を流し、USB給電の電圧が一瞬下がってボードがリセットすることがあります。

```text
サーボ（SG90の場合）
  茶（GND）  ── 外部5V電源の− と ボードのGND（共通に！）
  赤（5V）   ── 外部5V電源の＋（電池ボックス等）
  橙（信号） ── GPIO11
```

- **GNDの共通接続を忘れない**こと。外部電源のGNDとボードのGNDがつながっていないと、信号の基準がそろわずサーボは動きません
- SG90の信号入力は3.3Vロジックで一般に動作しますが、個体によっては不安定なことがあります
- サーボの赤い線をボードの3V3ピンにつなぐのも不可です（電流が足りず、他の回路も巻き込みます）

## 実行方法

```bash
cargo run --release
```

1秒ごとにサーボが左右へ首を振れば成功です（このコードは未検証のため、`configure`がエラーを返す場合は分解能や周波数の見直しが必要です）。

## よくある失敗

- **サーボが動くとボードが再起動する**: サーボの電源をUSBの5Vピンから取っています。起動電流で電圧が落ちています。外部電源＋GND共通に変えてください
- **サーボがぴくりとも動かない**: 外部電源とボードのGNDが共通になっていないことが最多です。信号は「GNDから見た電圧」なので、基準がそろわないと伝わりません
- **端でジジジと音がして熱くなる**: パルス幅が可動範囲の外を指していて、サーボが突き当たりに押し付け続けています。dutyを1段内側（6%や9%）へ戻してください
- **角度が中途半端にしか変わらない**: 整数パーセントの制約で、そもそも6段階しか指定できません。仕様どおりです

## やってみよう

`set_duty(5)`と`set_duty(10)`の間に`set_duty(7)`（中央付近）を挟み、3点を往復する動きにしてみましょう。待ち時間を300msに縮めると、サーボの追従の速さも観察できます。

## 確認問題

1. サーボの角度を決めるのは、周波数・duty比・パルス幅のどれですか。
2. パルス幅1.5msは、50Hzの信号ではduty何%ですか。
3. サーボの電源を外部5Vにした場合でも、ボードとつながなければならない線は何ですか。それはなぜですか。

<details>
<summary>答え</summary>

1. パルス幅（Highの時間の長さ）です。周波数は50Hz固定が標準で、duty比はパルス幅と周期から結果的に決まる値です。
2. 1.5ms ÷ 20ms = 7.5%です。`set_duty`は整数しか受け取れないため、7%か8%で近似することになります。
3. GNDです。信号線の電圧は「GNDを基準にした高さ」なので、両者のGNDが共通でないとサーボは信号を読み取れません。

</details>

## まとめ

- サーボは50Hz・パルス幅1〜2ms（目安）で角度が決まる。duty比ではなくパルス幅が命令
- LEDCの周波数を50Hzにすれば駆動できるが、整数パーセントのdutyでは6段階の粗い制御になる
- モーター電源は外部5V、GNDは必ずボードと共通。USB 5Vから取るとリセットの原因
