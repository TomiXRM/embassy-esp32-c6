## このページでできるようになること

- duty比・分解能・duty値（カウント）の3つの関係を計算できる
- ある範囲の値を別の範囲へ変換する式を整数演算で書ける
- 変換式に潜む3つの落とし穴（あふれ・切り捨て・範囲外）を説明できる

## 先に結論

duty比には3つの表現があります。**割合（50%）**、**時間（5kHzなら0.1ms）**、**カウント値（12bitなら2048）**です。どれも同じものを別の単位で見ているだけで、換算は掛け算と割り算でできます。esp-halの`set_duty`はパーセント（0〜100）を受け取るので、実際のプログラムでは「センサ値の範囲 → 0〜100」への**範囲変換**が中心になります。公式は**出力 = 入力 × 出力の最大値 ÷ 入力の最大値**。整数で書くときは「広い型で・掛けてから割り・上限を保証する」の3点を守ります。

## 身近なたとえ

100点満点のテストの点数を、5段階評価に直すことを考えてください。80点なら 80 × 5 ÷ 100 = 4です。満点どうしの比率で割り当てる——範囲変換はこれだけのことです。

ただしプログラムの整数演算では、計算の途中経過が「何点満点か」を常に意識する必要があります。掛け算の途中で型の上限を超えたり、割り算で小数点以下が消えたりするからです。数学のように順序を気にせず書けない点が、テストの点直しとの違いです。

## 仕組み

### duty比の3つの顔

5kHz・12bitのPWMでduty 50%を例にします。

| 表現 | 値 | 換算 |
|---|---|---|
| 割合 | 50% | 基準 |
| 時間 | 0.1ms | 周期0.2ms（5kHz）× 50% |
| カウント値 | 2048 | 4096段階 × 50% |

12bit分解能とは、1周期を4096カウントで刻むという意味でした。duty 50%はカウント2048に対応します。esp-halの`set_duty`はパーセントを受け取り、内部でこのカウント値への換算を行います。つまり私たちが書くのは「何かの値 → 0〜100%」の変換だけです。

### 範囲変換の公式

ADCの読み値（0〜4095）をdutyパーセント（0〜100）へ変換します。

**duty = 読み値 × 100 ÷ 4095**

一般形は「出力 = 入力 × 出力最大 ÷ 入力最大」です。逆向き（パーセント→カウント）も同じ形で書けます: カウント = duty × 4095 ÷ 100。

### 3つの落とし穴

1. **あふれ（オーバーフロー）**: 4095 × 100 = 409500は`u16`の上限65535を超えます。`u32`へ広げてから掛けます
2. **切り捨て**: 整数の割り算は小数点以下を捨てます。先に割ると`4094 / 4095 = 0`のように情報が全部消えるため、必ず掛けてから割ります。それでも最後の割り算で最大1未満の誤差は残ります（duty換算なら1%未満で、実用上は問題ありません）
3. **範囲外の入力**: 校正済みADCはまれに4095を超える値を返します。変換結果が100を超えると`set_duty`は`Err`を返すので、`.min(100)`で上限を保証します

## RustとEmbassyではどう書くか

`examples/13-adc-pwm`の変換行がこの公式の実装です（抜粋）。

```rust
// ADCの読み値(0〜4095)をデューティ比(0〜100%)に変換する
// 校正の補正でわずかに4095を超えることがあるためmin(100)で上限を保証
let duty_pct = ((raw as u32 * 100) / 4095).min(100) as u8;
channel0.set_duty(duty_pct).unwrap();
```

## コードを一行ずつ読む

```rust
let duty_pct = ((raw as u32 * 100) / 4095).min(100) as u8;
```

内側から順に読みます。

1. `raw as u32` — 落とし穴1への対策。掛け算の前に広い型へ
2. `* 100 / 4095` — 公式そのもの。落とし穴2への対策として、この順序で
3. `.min(100)` — 落とし穴3への対策。上限を保証
4. `as u8` — `set_duty`の引数型`u8`へ戻します。`.min(100)`のあとなので、この縮小変換で値が壊れることはありません

この1行は「範囲変換の定石」として、ADC以外（温度→ファン速度、距離→音量など）にもそのまま応用できます。

## 実行方法

`examples/13-adc-pwm`で確認できます。シリアル出力の生値とdutyが、公式どおりの関係になっていることを見てください。

```bash
cargo run --release
```

```text
INFO - ADC生値 = 2051, PWMデューティ =  50%   ← 2051×100÷4095 = 50.08 → 50
INFO - ADC生値 =  409, PWMデューティ =  10%   ← 409×100÷4095 = 9.98 → 9か10付近
```

## よくある失敗

- **`(raw / 4095) * 100`と書いて常に0%**: 先に割ったため切り捨てで0になっています。掛けてから割ります
- **`raw * 100`で`attempt to multiply with overflow`**: `u16`のまま掛けています。デバッグビルドではパニック、リリースビルドでは黙って値が壊れる（wrapする）ため、より発見しにくくなります。型を広げるのが正解です
- **4095と4096を取り違える**: 段階の数は4096、最大の読み値は4095です。割る数は「入力の最大値」なので4095を使います（4096で割っても誤差はわずかですが、最大値を入れても100にならない式になります）
- **範囲変換を小数（f32）で書く**: 動きますが、ESP32-C6には浮動小数点の専用回路がなく低速です。整数で書けるものは整数で書きます

## やってみよう

変換先を0〜100ではなく「30〜100」にしてみましょう（LEDが完全には消えない調光）。公式を2段使います: まず0〜70へ変換し、30を足します。`let duty = (raw as u32 * 70 / 4095 + 30).min(100) as u8;`のような形になります。つまみを絞ってもLEDがうっすら光り続ければ成功です。

## 確認問題

1. 12bit・5kHzのPWMで、duty 25%はカウント値とHigh時間でそれぞれいくつですか。
2. `raw * 100 / 4095`の`raw`が`u16`のままだと何が起きますか。デバッグビルドとリリースビルドの違いも述べてください。
3. `.min(100)`が防いでいるのは、どこから来るどんな値ですか。

<details>
<summary>答え</summary>

1. カウント値は4096 × 0.25 = 1024、High時間は周期0.2ms × 0.25 = 0.05ms（50µs）です。
2. 4095 × 100 = 409500が`u16`の上限65535を超えてあふれます。デバッグビルドではパニックし、リリースビルドでは値が折り返されて（wrapして）誤った結果のまま進みます。
3. `AdcCalBasic`の校正補正によってADCがまれに返す、4095をわずかに超える読み値です。変換結果が101以上になると`set_duty`が`Err`を返すため、事前に100で頭打ちにします。

</details>

## まとめ

- duty比は割合・時間・カウント値の3表現。換算は掛け算と割り算だけ
- 範囲変換の公式は「入力 × 出力最大 ÷ 入力最大」。広い型で、掛けてから割り、上限を保証する
- この定石はADC→PWMに限らず、あらゆる値のスケーリングに使い回せる
