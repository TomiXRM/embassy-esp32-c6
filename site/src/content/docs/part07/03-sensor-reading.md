## このページでできるようになること

- ADC生値を「電圧」や「パーセント」などの意味のある値に変換できる
- ノイズで揺れる読み値を移動平均でならせる
- 小数を使わず整数演算で変換する理由と書き方が分かる

## 先に結論

ADCが返すのは0〜4095の**生値**で、そのままでは意味が読み取れません。「生値 → 電圧 → 物理量」の順に変換して初めて使える値になります。またADCの読み値はノイズで常に少し揺れるため、直近の数回分を平均する**移動平均**でならすのが定石です。変換は`raw * 3300 / 4095`のように**mV単位の整数演算**で書くと、小数(浮動小数点)を避けられて速く確実です。

このページの移動平均コードは検証済みexampleにはない追加コードです（構文レビュー済み、実機未検証）。

## 身近なたとえ

テストの点数を思い浮かべてください。「合計点312点」と言われてもピンときませんが、「500点満点中312点、つまり62%」なら意味が分かります。生値から物理量への変換は、この「満点で割って意味のある単位にする」操作です。

また、体重計に乗るたびに表示が数百グラム揺れるとき、3回測って平均するのと同じ発想が移動平均です。ただし移動平均は「新しい測定のたびに、直近N回の平均を出し続ける」点が、1回きりの平均と違います。

## 仕組み

### 変換の3段階

```mermaid
graph LR
  A[生値<br>0〜4095] -->|×3300÷4095| B[電圧 mV<br>0〜3300]
  B -->|センサの変換式| C[物理量<br>温度・角度・%など]
```

1. **生値**: ADCが返す0〜4095
2. **電圧**: 生値 ÷ 4095 × 3.3V。プログラムではmV単位の整数で `raw * 3300 / 4095`
3. **物理量**: センサのデータシートにある式で電圧を温度などへ変換。可変抵抗なら「つまみ位置0〜100%」が物理量にあたります

### なぜ整数演算か

ESP32-C6のCPU（RISC-V RV32IMAC）は浮動小数点専用の計算回路を持たないため、`f32`の計算はソフトウェアで肩代わりされて遅くなります。電圧を「3.3Vの小数」ではなく「3300mVの整数」で持てば、精度を保ったまま整数だけで計算できます。

計算の順序に注意してください。`raw * 3300`を先に計算すると最大 4095 × 3300 = 約1351万で、`u16`（最大65535)からあふれます。先に`u32`へ広げてから掛けます。

### 移動平均

ADCの読み値は、電源の揺れや配線が拾う電気的な雑音（**ノイズ**）で毎回少しずつ変わります。**移動平均**は「直近N回の読み値の平均」を答えとする方法で、揺れがおよそ1/√N に小さくなります。N=8なら揺れは約1/2.8です。

代償もあります。平均する分だけ、本当の変化への反応が遅れます。Nを大きくするほど滑らかになりますが、つまみを回してから値が追いつくまでの時間も伸びます。滑らかさと反応速度のトレードオフです。

## RustとEmbassyではどう書くか

リングバッファ（固定長配列を循環して使う方法）で移動平均を実装します。**このコードは検証済みexampleにはない追加例で、cargo checkを通していません**。読み取り部分（`read_oneshot`）は`examples/13-adc-pwm`と同じです。

```rust
const N: usize = 8; // 平均する回数

let mut samples = [0u16; N]; // 直近N回の生値
let mut index = 0;

loop {
    // 1回読んで、いちばん古い値と入れ替える
    samples[index] = adc1.read_oneshot(&mut pot_pin).await;
    index = (index + 1) % N;

    // 合計はu32で計算（u16のままだと4095×8であふれる可能性）
    let sum: u32 = samples.iter().map(|&s| s as u32).sum();
    let avg = sum / N as u32;

    // 生値の平均 → 電圧(mV) → つまみ位置(%)
    let millivolts = avg * 3300 / 4095;
    let percent = avg * 100 / 4095;

    info!("平均生値 = {avg:4}, 電圧 = {millivolts:4}mV, 位置 = {percent:3}%");

    Timer::after(Duration::from_millis(100)).await;
}
```

## コードを一行ずつ読む

```rust
let mut samples = [0u16; N];
let mut index = 0;
```

- 固定長配列をバッファにします。組み込みではヒープを使わないので、サイズはコンパイル時に決めます（[第5部 5. heapを使わない設計](/embassy-esp32-c6/part05/05-heap/)）

```rust
samples[index] = adc1.read_oneshot(&mut pot_pin).await;
index = (index + 1) % N;
```

- 書き込み位置を1つ進め、`% N`で末尾まで来たら先頭へ戻します。これで配列が「直近N回分」を常に保持します

```rust
let sum: u32 = samples.iter().map(|&s| s as u32).sum();
```

- 合計を`u32`で計算します。`u16`のまま足すと 4095 × 8 = 32760 はぎりぎり収まりますが、Nを増やした瞬間にあふれます。最初から広い型で計算するのが安全です

```rust
let millivolts = avg * 3300 / 4095;
```

- 「掛けてから割る」順序が大切です。先に`avg / 4095`を計算すると、整数の割り算なのでほぼ常に0になってしまいます

## 実行方法

`examples/13-adc-pwm`をベースに、ループ部分を上のコードへ書き換えて実行します。

```bash
cargo run --release
```

つまみを固定したとき、平均前より表示の揺れが小さくなること、つまみを速く回したとき値が少し遅れて追いつくことを確認してください。

## よくある失敗

- **平均がいつも0になる**: `avg / 4095 * 3300`のように先に割っています。整数の割り算は小数点以下を切り捨てるので、4095未満 ÷ 4095 = 0です。必ず先に掛けます
- **`attempt to multiply with overflow`でパニックする**: `u16`のまま`raw * 3300`を計算すると65535を超えます。`as u32`で広げてから掛けてください
- **起動直後の値が低く出る**: バッファの初期値0が平均に混ざるためです。最初のN回は表示しない、または起動時にN回読んでバッファを埋めてから始めると解決します
- **平均しても大きく揺れる**: 配線の接触不良やワイパー端子の浮きが原因のことが多いです。平均はノイズをならせますが、接続の問題は隠せません

## やってみよう

`N`を8から32に増やして、表示の滑らかさと反応の遅れがどう変わるか比べてみましょう。逆にN=2ではほとんど平均の効果がないことも確認できます。

## 確認問題

1. 生値2048を電圧(mV)に変換する式と、その結果を書いてください。
2. `avg / 4095 * 3300`という式がほぼ常に0になるのはなぜですか。
3. 移動平均のNを大きくすると、良くなることと悪くなることは何ですか。

<details>
<summary>答え</summary>

1. 2048 × 3300 ÷ 4095 ≒ 1650mV（約1.65V）です。
2. 整数の割り算は小数点以下を切り捨てるからです。avgが4095未満なら`avg / 4095`は0になり、その後に何を掛けても0のままです。
3. 良くなること: ノイズによる揺れが小さくなる（約1/√N）。悪くなること: 本当の変化に値が追いつくまでの遅れが大きくなる。メモリ使用量もN分増えます。

</details>

## まとめ

- 変換は「生値 → 電圧(mV) → 物理量」の順。mV整数で「掛けてから割る」
- 移動平均は直近N回の平均。揺れは約1/√Nに減るが、反応は遅れる
- 合計や積は`u32`など広い型で。あふれと切り捨てが整数演算の2大注意点
