## このページでできるようになること

- 用途に応じたPWM周波数の目安（LED・サーボ・モーター）を挙げられる
- 「周波数を上げると分解能が下がる」トレードオフを式で説明できる
- 教材のLED調光が「5kHz・12bit」である理由を説明できる

## 先に結論

PWM周波数は自由に選べるようで、実は両側から挟まれています。低すぎるとLEDのちらつきやモーターの唸りが出て、高すぎるとdutyの**分解能**（明るさを刻める細かさ）が足りなくなります。分解能Nビットと周波数fの間には**2のN乗 × f ≦ 元クロックの周波数**という制約があるからです。教材の5kHz・12bitはこの制約内の実用的な組み合わせで、同じ5kHzでも13bitは制約を超えるため設定できません。サーボだけは例外で、規格として50Hz固定です。

## 身近なたとえ

1分間で1枚の紙を等分に折って冊子を作るとします。ページ数（分解能）を増やすほど1ページあたりに使える時間は短くなり、いつか折る時間が足りなくなります。PWMも同じで、1周期の中をdutyの段階数だけ刻む必要があり、周期が短い（周波数が高い）ほど1段階に使える時間が短くなります。

実際のLEDCでは「1段階の時間」の最小値は元クロックの1周期で決まっていて、これ以上は縮められません。ここが限界の正体です。

## 仕組み

### 分解能と周波数のトレードオフ

LEDCのタイマーは、元クロック（この教材ではAPBクロック = 40MHz）を数えてPWMの1周期を作ります。Nビット分解能なら1周期を2のN乗個に刻むので、

**2のN乗 × PWM周波数 ≦ 40MHz**

が成り立つ必要があります（実際には元クロックを割り算する分周器の制約も加わります）。表にすると次のとおりです。

| PWM周波数 | 40MHz ÷ f | 選べる最大分解能の目安 |
|---|---|---|
| 50Hz | 800000 | 十分に高い（サーボはこの上限には当たらない） |
| 1kHz | 40000 | 15bit相当まで（2^15 = 32768） |
| 5kHz | 8000 | **12bit（2^12 = 4096 ≦ 8000、13bitの8192は超過）** |
| 40kHz | 1000 | 9bit（2^9 = 512） |
| 1MHz | 40 | 5bit（2^5 = 32、つまり32段階しかない） |

教材のコードで13bit・5kHzを指定すると`configure`が`Err`を返すのは、まさにこの制約のためです。8192段階 × 5kHz = 40.96MHzとなり、40MHzのクロックでは1段階分の時間を作れません。

### 低すぎる周波数の問題

- **ちらつき**: 人間の目はおよそ100Hz以下の点滅を知覚します。LEDの調光は数百Hz以上、スマホカメラ越しの縞模様も避けるなら1kHz以上が安心です
- **音**: 20Hz〜20kHzは人間の可聴域です。モーターやコイルをこの範囲のPWMで駆動すると、部品自体が振動してピーという音を出すことがあります。モーター駆動で20kHz以上がよく使われるのは、音を可聴域の外へ出すためです

### 用途別の目安

| 用途 | 周波数の目安 | 理由 |
|---|---|---|
| LED調光 | 1k〜20kHz | ちらつき回避。教材は5kHz |
| サーボ | **50Hz固定** | 規格。パルス幅が命令（[前ページ](/embassy-esp32-c6/part07/06-servo/)） |
| DCモーター | 20kHz以上が多い | 可聴音の回避（本格的にはMCPWMの領分） |

## RustとEmbassyではどう書くか

周波数と分解能を決めているのはタイマー設定の2行です（`examples/13-adc-pwm`より抜粋）。

```rust
lstimer0
    .configure(timer::config::Config {
        duty: timer::config::Duty::Duty12Bit,   // 分解能: 4096段階
        clock_source: timer::LSClockSource::APBClk,
        frequency: Rate::from_khz(5),           // 周波数: 5kHz
    })
    .unwrap();
```

`configure`が組み合わせの成立を検査してくれるので、「動かして初めて破綻に気づく」ことはありません。実現不能な組み合わせは`Err`で返ります。

## 実行方法

`examples/13-adc-pwm`で、`Rate::from_khz(5)`を`Rate::from_hz(200)`などに変えて書き込み、LEDを軽く視線を動かしながら見てください。低い周波数では点滅の尾が見えます。

```bash
cargo run --release
```

## よくある失敗

- **`Duty13Bit`・5kHzで`configure`がエラー**: 2^13 × 5kHz = 40.96MHz > 40MHzで、制約を超えています。12bitに下げるか周波数を下げます
- **周波数を上げたのに明るさの段階が粗くなった**: トレードオフそのものです。高周波数では取れる分解能が下がります。用途に必要な段階数から逆算して周波数を決めます
- **モーターを1kHzのPWMで回したら鳴く**: 1kHzは可聴域のほぼ中心です。周波数を可聴域の上（20kHz超）へ移すと消えますが、その分だけ分解能は下がります
- **サーボの周波数を「性能向上のため」に上げる**: サーボは50Hzを前提に設計されています。勝手に変えると誤動作や故障の原因になります

## やってみよう

`Rate::from_hz(50)`（LEDのまま）にして、LEDを観察してみましょう。50Hzでも正面からじっと見るとほぼ連続に見えますが、視線をさっと動かすと点線状に見えます。目の残像とちらつきの関係を体感できます。

## 確認問題

1. 40MHzのクロックで20kHzのPWMを作るとき、選べる最大の分解能は何bitですか。
2. 教材のLED調光で13bit・5kHzが設定できない理由を式で説明してください。
3. DCモーターのPWMに20kHz以上がよく使われる理由は何ですか。

<details>
<summary>答え</summary>

1. 40MHz ÷ 20kHz = 2000。2^10 = 1024 ≦ 2000 < 2048 = 2^11なので、最大10bitです。
2. 2^13 × 5kHz = 8192 × 5000 = 40.96MHzとなり、元クロック40MHzを超えるためです。1周期を8192段階に刻む時間が確保できません。
3. 20Hz〜20kHzの可聴域を避け、部品の振動音（鳴き）を人間に聞こえなくするためです。

</details>

## まとめ

- 制約は「2^N × 周波数 ≦ 元クロック」。周波数と分解能はトレードオフ
- 低すぎるとちらつき・鳴き、高すぎると分解能不足。LEDは1k〜20kHz、サーボは50Hz固定
- 教材の5kHz・12bitは制約内の実用解。13bit・5kHzは`configure`が`Err`で教えてくれる
