## このページでできるようになること

- 実戦投入されたファームウェアの「規模の現実」（コード量・品質管理・妥協）を具体的に言える
- この応用編で読んだ設計のうち、ESP32-C6へそのまま持ち帰れるものと読み替えが必要なものを区別できる
- 自分のプロジェクトで次にやる演習を2つ持っている

## 先に結論

luhbots（ドイツ・ハノーファーのLeibniz Universität Hannoverの学生チーム）は、このファームウェアでRoboCup SSLの世界大会を戦い、2022年にDiv B 3位、2023年にDiv B 準優勝、2024年にはDiv Aで5位タイという成績を残しました。そのコードはRust約17,400行。clippyの最も厳しい警告設定で磨かれ、CIが回り続ける一方で、READMEは空で、応急処置のコメントも残っています。**実戦のコードとは、理想と妥協が同居したまま結果を出すもの**です。そして大事なのはここからです——この4基板のロボットを支えている部品の多く（Observable、1kHzのTicker、UARTプロトコル、Watchdog、ADCのヒステリシス）は、**あなたがこの教材で学んだものと同じ**です。このページで対応表を確認し、演習2本で自分の手に移して、教材を締めくくります。

## 身近なたとえ

プロの厨房を見学した後の気持ちに似ています。鍋は業務用で火力も桁違いですが、やっていることは「切る・加熱する・味付けする」——家庭の台所と同じ基本動作の組み合わせです。見学の価値は「同じ基本でここまで行ける」と知ることと、「業務用ならではの工夫」を持ち帰ることにあります。

たとえと違うのは、ソフトウェアでは「業務用の道具」まで無料で持ち帰れることです。MITライセンスのこのコードは、読むだけでなく、部品を自分のプロジェクトに移植することまで許されています（出典とライセンス表記は忘れずに）。

## 規模の現実 — 数字と、正直な中身

出典は [luhbots/luhsoccer_firmware](https://github.com/luhbots/luhsoccer_firmware)（MITライセンス）と、チームが公開している2023年のTDP（Team Description Paper、技術説明書）です。

| 項目 | 実際 |
|---|---|
| コード量 | Rust約17,400行（4基板+共有ライブラリ。取り込んだHAL除く） |
| 静的検査 | clippyの`pedantic`+`nursery`（最も口うるさい警告群）を有効化 |
| CI | GitLab CIでclippy・テストカバレッジ・ベンチマークを自動実行 |
| テスト | **libs（共有ライブラリ）のみ単体テストあり**。ファームウェア本体は`test = false`（実機がないと意味のあるテストにならないため） |
| ドキュメント | READMEは空。コメントアウトされた残骸も残る |
| 妥協の痕跡 | 「基板設計者がオペアンプを正帰還にしてしまった」「This is a hotfix」等のコメント（9ページ） |

この表は上下セットで読んでください。上半分だけなら「意識の高い理想のプロジェクト」、下半分だけなら「雑なコード」に見えますが、実際は両方が同時に真実です。**検査とCIで守れるところは機械に守らせ、実機依存のテストや書類仕事は割り切り、ハードの不具合は記録してソフトで回避する**。限られた学生の時間で世界大会を戦うための、資源配分の結果です。

ちなみに彼らがRustを選んだ理由は、2023年TDPに書かれています。要旨は「マイコン変更に伴いファームウェアを全面的に書き直した。将来またマイコンが変わっても対応できるよう、Rustとembedded-halクレートでアーキテクチャ間のコード再利用を可能にした」。**チップが変わってもコードを持ち越す**という狙いは、この教材が「esp-halの下にembedded-halがある」と繰り返し説明してきた理由と同じです。

## C6教材への対応表

この応用編で読んだ部品を、「そのまま持ち帰れるもの」と「C6では読み替えるもの」に仕分けします。

### そのまま持ち帰れるもの

| ロボットでの姿 | 教材での対応 | 備考 |
|---|---|---|
| Observable（131行の自作同期プリミティブ、6ページ） | 第9部9 Channel/Signal/Mutex | 依存はembassy-syncだけ。C6でもそのまま書ける（下の演習1） |
| 1kHz Ticker+lost wakeup検出（8ページ） | 第6部8 Ticker、第6部9 with_timeout | `with_timeout(ticker.next())`で「周期を取りこぼした」ことを検出する技 |
| UARTプロトコル postcard+COBS+CRC16（7ページ） | 第12部10 最終プロジェクトのprotocol.rs | フレーミング+誤り検出の考え方は同一。共有クレート化(intra-comms)が実戦版 |
| Watchdog給餌task（10ページ） | 第6部10 Watchdog | 「無条件給餌はexecutor凍結の検出」という限界の理解ごと持ち帰る |
| ADC+ヒステリシス状態機械（9ページ） | 第8部 ADC、第4部9 状態機械 | 電池監視・センサ閾値の定番構造 |
| フラッシュへの設定保存 | 第12部5 | 設定変更+SAVEシグナルの構造も同じ |
| featureフラグでのテストtask注入（11ページ） | Cargoの標準機能 | C6プロジェクトで今日から使える |

### C6では読み替えるもの

| ロボット（RP2040/ATSAM） | ESP32-C6での読み替え |
|---|---|
| PIO（プログラマブルI/O。DAC・ブザー・WS2812制御に使用） | RMT（WS2812）やLEDC（PWM）など、専用ペリフェラルで置き換える |
| 2コア（制御ループがcore1を専有） | C6は1コア。専有コアの代わりに優先度（高優先実行器）で守る |
| InterruptExecutor（SWI_IRQ_0） | esp-rtosの相当機能（高優先の実行器）で同じ設計ができる |
| SX1280無線+protobuf基地局 | Wi-FiやESP-NOWに概念ごと写像できる（第11部）。「RXタイムアウトでゼロ化」の設計はそのまま有効 |

読み替えの共通原則はこうです——**「何をしたいか（周期制御を守る、無線断で止まる）」は移植でき、「どのハードでやるか」は差し替える**。TDPの言うRustの狙いどおりです。

## 演習1: Observableを自分のC6プロジェクトで書く

6ページで読んだObservableは、依存がembassy-syncだけの純粋なライブラリコードでした。つまり**C6のプロジェクトでそのまま自作できます**（この教材のスタックはembassy-sync 0.7です）。

やること:

1. 新しいモジュール`observable.rs`を作り、「現在値を持つ」「`get()`でいつでも読める」「`set()`で更新すると購読者が起きる」「購読者は`next_value().await`で変化を待てる」の4つを満たす型を設計する
2. まず`Mutex`（blocking版）で現在値を持ち、通知は購読者ごとの`Signal`や、embassy-syncのWatch/PubSubの仕組みを参考に組む。**最初は購読者1人の`Observable`でよい**——それなら現在値+`Signal`の組み合わせで書けます
3. 動作確認: 送り手taskが1秒ごとにカウンタを`set()`し、受け手taskが`next_value().await`でログを出す。**受け手が遅くても最新値だけが読める**（Channelとの違い）ことを確認する
4. 余力があれば、原作のように`set_if_different()`（値が変わったときだけ通知）を足し、なぜこれがあると通信量が減るのか（7ページのUART再送を思い出す）を自分の言葉で書く

原作の`libs/sync/src/observable.rs`（131行）と答え合わせをすれば、「読んだコードが書けるコードになる」体験ができます。

## 演習2: 最終プロジェクトにフェイルセーフ層を1つ足す

第12部10の無線ボタン端末（examples/final-wireless-button）には、リトライ・縮退・自動復帰(第12部7)はありますが、10ページで見た多層フェイルセーフの発想で見直すと、まだ足せる層があります。次のどれか1つを選んで実装してください。

- **受信側にRXタイムアウト層**: ロボットの①と同じ発想で、「最後の受信から一定時間たったら表示を安全側（消灯や警告表示）に落とす」。`with_timeout`だけで書けます
- **キープアライブ層**: ロボットの③と同じ発想で、送信側が「変化がなくても最低1Hzで現状を再送」する。受信側のタイムアウト層と対になり、切断検出が確実になります
- **自己監視層**: メインループが一周するたびにフラグを立て、別taskが「一定時間フラグが立たない=詰まった」を検出してログと表示を出す。無条件給餌のWDTでは検出できない異常（10ページ）を自分で見張る練習です

どれを選んでも、実装は数十行です。大事なのは実装量ではなく、**「この層はどの区間のどんな故障を守るのか」を先に書いてから実装する**こと。10ページの表の1行を、自分のプロジェクトに増やす感覚です。

## 3つの応用編を締めくくる

| 応用編 | 題材 | 中心テーマ |
|---|---|---|
| キーボード | 自作キーボードファームウェア | 入力装置・USB/BLE・核とドライバの分離 |
| センサ端末 | 電池駆動の気象ステーション | 省電力・Deep Sleep・エラー設計 |
| ロボット | RoboCup SSLの4基板ファームウェア | 分散・実時間制御・安全設計・拡張性 |

3つに共通させたのは「**実物を読む**」という方法でした。教材のサンプルは学ぶために作られていますが、実物は勝つため・売るため・動き続けるために作られています。それでも読めたのは、部品が同じだったからです。task、Channel、Ticker、所有権、状態機械——第1部から積み上げた道具は、17,000行の世界チャンピオンではないが世界と戦うコードの中で、そのまま使われていました。

**実物が読めれば、作れる。** 読めるということは、設計の意図と妥協が見分けられるということです。次にあなたが作るものは、教材の写経ではなく、読んだ実物たちから部品と規律を選び取った、あなたの設計になります。

## よくある誤解

- **「17,000行は自分には一生無理」**: この応用編は12ページで読み切りました。読めたのは、行数ではなく構造（spawnリスト、Observableの配線、層の分担)をたどったからです。書くときも同じで、正しい構造を保てば行数は後から増えても壊れません
- **「実戦コードには単体テストが完備されているはず」**: このプロジェクトのテストはlibsのみで、ファーム本体は`test = false`です。実機が絡む部分のテストは費用対効果が悪いという判断で、代わりにclippy・CI・型・featureフラグのテストビルドで品質を守っています。テストは目的ではなく品質戦略の一手段です
- **「強豪のコードだから真似ればすべて正しい」**: 空のREADMEや応急処置コメントが示すとおり、実物は模範解答ではありません。持ち帰るべきは個々の行ではなく、「何を機械に守らせ、何を割り切ったか」という判断のしかたです

## 確認問題

1. このプロジェクトで単体テストがlibsにしかないのは手抜きと言えるでしょうか。彼らの品質管理の全体像を挙げて答えてください。

<details>
<summary>答え</summary>

単純な手抜きとは言えません。ファーム本体は実機（モータ、無線、230Vのキッカー）がないと意味のある自動テストが難しいため`test = false`とし、その代わりにclippy pedantic+nurseryの静的検査、CIでのカバレッジ・ベンチマーク、型付きの通信による設計上の防御、featureフラグの実機テストビルド（test_dribbler等）で品質を守っています。テスト可能な純粋ロジック（libs）にはテストを書いており、資源配分の判断として一貫しています。

</details>

2. 「1kHz制御ループをcore1に専有させる」設計をESP32-C6へ持ち込むとき、何をどう読み替えますか。

<details>
<summary>答え</summary>

C6は1コアなので専有コアは使えません。守りたいのは「制御ループの周期が他の処理に乱されないこと」なので、esp-rtosの高優先実行器（InterruptExecutor相当）へ制御taskを置き、優先度で周期を守る形に読み替えます。さらにwith_timeoutでticker.next()を包めば、周期を取りこぼしたこと自体を検出できます。

</details>

3. この応用編から自分のプロジェクトに最初に持ち帰るとしたら何を選びますか。理由も述べてください。

<details>
<summary>答え</summary>

（自由回答）例: 「RXタイムアウトで安全側に落とす」層。数十行で書けるのに、無線が絡むどんなプロジェクトでも通信断という最も起きやすい故障を安全化できるため。ほかにObservable、キープアライブ、featureフラグのテストtask注入なども好例です。

</details>

## まとめ

- 実戦の17,400行は、厳格な静的検査・CIと、空のREADME・応急処置が同居する「資源配分の結果」。理想化も軽蔑もせず、判断ごと読むのが実物読解
