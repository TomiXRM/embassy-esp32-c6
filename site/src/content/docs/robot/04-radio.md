## このページでできるようになること

- SSLルールが無線に課す制約（申告制・2キャリア・Bluetooth禁止）とその理由を説明できる
- SX1280＋FLRC＋同期語によるリンクの仕組みと、rf_taskの動作を読める
- 「受信50ms途絶で全出力ゼロ」というフェイルセーフを、教材のハートビート設計と対比して説明できる
- 2022年のESP32＋NRF24L01構成が置き換えられた経緯から、無線設計の教訓を読み取れる

## 先に結論

ロボットの命綱である無線リンクは、Semtech **SX1280**という2.4GHz無線チップの**FLRC**という変調方式（1.3Mb/s）で作られています。宛先の区別は**ロボットごとに違う32bitの同期語**で行い、無線チップ自体が「自分宛て以外」を捨てます。送信出力はSKY66112というアンプで増強——この構成は強豪TIGERs Mannheimに倣ったもので、その前の年のNRF24L01構成が**本番で通信に失敗した**教訓から選ばれました。そしてこのページの主役はロボット側rf_taskの1行です: **50ミリ秒受信がなければ、速度・キック・ドリブラーをすべてゼロにする**。教材の最終プロジェクトで作ったハートビート途絶検出の、実戦版がここにあります。

## 身近なたとえ

この無線リンクは、**先生（基地局）が出席簿順に生徒（ロボット）を次々あてていく教室**です。先生が「1番、指示はこれ。状況を報告して」と言うと、1番の生徒だけが即座に答えます。ほかの生徒には聞こえていても、自分の番号でなければ黙っています。全員あて終わったらまた最初から。生徒側のルールは1つ、「**しばらく先生の声が聞こえなくなったら、その場で止まって待て**」です。

——ただし実際の技術では、「自分の番号でなければ黙る」判断は生徒（マイコン）ではなく耳（無線チップ）がやります。同期語が一致しないパケットはチップが捨てるので、マイコンは自分宛てのときしか起こされません。

## SSLルールが無線に課すもの

公式ルールの無線の節（3.2.6 Radio Communication）は、技術的な制約を明確に定めています。

- 無線の**方式・出力・周波数を運営委員会に申告**すること。変更時も速やかに届け出る
- 干渉を避けるため、試合前に**2つの搬送波周波数から選べる**こと
- **Bluetoothの使用は禁止**。理由もルールに明記されています: 「周波数チャネルを固定できないため」（"Bluetooth is not allowed since it cannot be fixed to frequency channels."）

Bluetoothは、混雑した電波環境で生き残るために**周波数ホッピング**——使う周波数を毎秒何百回も切り替える仕組み——を使います。日常の機器には長所ですが、大会会場では「どの周波数を使うか申告し、他チームと調整する」ことができない無線になってしまいます。だから禁止。**便利な機能が、環境が変わると失格要件になる**面白い例です。

逆に言うと、ロボット用の無線には「周波数を固定でき、申告でき、必要なら第2の周波数に切り替えられる」性質が求められます。rf_taskのコードには実際に、**動作中に周波数変更の指示を受けて設定し直す**処理が入っています（後述のselect3の分岐です）。ルールの要求がそのままコードの形になっている例です。

## SX1280というチップ、FLRCという変調

SX1280はSemtech社の2.4GHz帯トランシーバです。LoRa（遠距離・低速）でも知られるチップですが、このロボットが使うのは**FLRC**（Fast Long Range Communication）というモードで、感度を稼ぎながら**1.3Mb/s**まで出せる、いわば「速さと届きやすさの中間」です。設定はロボット側・基地局側で対になっています。

```rust
// 出典: luhsoccer_firmware (luhbots, MIT) maincontroller/src/rf.rs（基地局側main.rsにも同じ設定）
sx.set_modulation_params(
    FlrcBitrateBandwidth::Bitrate1300Bandwidth12,  // 1.3Mb/s
    FlrcCodingRate::CodingRate11,
    FlrcModulationShaping::BtOff,
)
```

宛先の区別が独特です。IPアドレスもMACアドレスもない代わりに、**同期語（sync word）**という32bitの値を使います。受信側チップは、パケット先頭の同期語が自分の設定値と一致したときだけ受信を続けます。共有クレートには16台分の同期語がずらりと定義されています。

```rust
// 出典: luhsoccer_firmware (luhbots, MIT) libs/intra-comms/src/lib.rs
pub const ROBOT_BLUE_SYNC_WORDS: [u32; 16] = [
    0xab60_615e,
    0x1902_97ab,
    0x56e2_4fb8,
    // ...（16台分続く）
];
```

基地局はロボット3番に送るとき、送信前に自分の同期語をROBOT_BLUE_SYNC_WORDS[3]へ切り替えます。ロボット3番だけがこのパケットで起こされ、他の15台のマイコンは割り込みすら受けません。**アドレス振り分けをハードウェアに任せる**——第10部でTWAIのハードウェアフィルタを学んだときと同じ発想が、無線でも使われています。

アンテナの前には**SKY66112**というPA/LNA（送信増幅器/低雑音受信増幅器）が入っています。2023年TDPは採用理由を率直に書いています: 強豪TIGERs Mannheimが同じSX1280＋SKY66112の組み合わせをバンコク大会で使い、接続問題を克服していたので、それに倣った、と。**動いている実績のある構成を採る**のも、立派な設計判断です。

## 2022年の失敗 — ESP32＋NRF24L01

なぜここまで無線に本気なのか。2023年TDPには前年の苦い経験が書かれています。

2022年の機体は、本来のメイン基板がハードウェア不良で使えず、**ESP32と市販モジュール（NRF24L01）による暫定構成**で大会に出ました。TDPいわく、NRF24L01はRoboCup 2022の会場で通信に問題を抱え、**サーバからロボットへは大きな損失込みでしか届かず、ロボットからの状態報告は返せなかった**——つまりテレメトリなしの片方向で戦ったのです。この経験から2023年に基板を新設計し、無線もSX1280へ置き換えました。

大会会場は、数百人がスマホを持ち、全チームが2.4GHz帯を使う、**家や研究室とは別世界の電波環境**です。「手元では動いた」が通用しない。無線モジュールの選定は仕様表ではなく**混雑環境での実績**で行う——高い授業料の教訓です。ちなみにこの教材のESP32-C6も2.4GHz帯で戦う仲間なので、Wi-FiやESP-NOWを混雑環境に持ち込むときには同じ心構えが要ります。

## rf_taskを読む — 受信、配布、応答

ロボット側の無線はメイン基板のrf_taskが担います。動作は1周期がきれいな3拍子です。

1. **受信を開始する**（タイムアウト付き）
2. 受信できたら、指令を検証して**Observableへ配布する**（速度指令、キック指令、ドリブラー速度…。Observableはこのファームウェア自作の同期プリミティブで、6ページ目の主役です。今は「最新値を配る掲示板」と思ってください）
3. **テレメトリを1発返す**（実測速度、ボール在否、キッカー電圧など）。共有クレートのコメントに「メインコントローラは、パケットを受信した後にのみ基地局へ1パケットを送る」と明記されています。基地局の「送って→短い窓で応答を待つ」ポーリング（前ページ)と、きっちり噛み合う設計です

そして受信開始の行に、このページのタイトルの数字がいます。

```rust
// 出典: luhsoccer_firmware (luhbots, MIT) maincontroller/src/rf.rs（抜粋）
sx.start_receive_packet_slice(
    32,
    PeriodBase::MilliSeconds1,
    if no_connection.get() { 0 } else { 50 },  // 接続中は50ms、切断中は無期限
    &mut encode_buffer,
    &mut decode_buffer,
)
```

接続が生きている間、受信は**50ミリ秒のタイムアウト付き**です（切断中は0＝タイムアウトなしで、次のパケットをじっと待ちます）。タイムアウトはSX1280チップ自身が数え、時間切れになると割り込みで知らせてきます。そのときの処理がこれです。

```rust
// 出典: luhsoccer_firmware (luhbots, MIT) maincontroller/src/rf.rs（抜粋）
if irq.is_set(IrqBit::RxTxTimeout) {
    warn!("timeout while receiving packet");
    command_velocity.set(LocalVelocity {
        forward: 0,
        left: 0,
        counterclockwise: 0,
    });
    command_kick_speed.set(crate::KickSpeed::Velocity(0));
    dribbler_speed.set(0);
    sky_outer = Some(sky.into_sleep_mode2());
    no_connection.set(true);
    continue;
}
```

50ミリ秒、基地局の声が聞こえなかったら——**走行速度ゼロ、キックゼロ、ドリブラーゼロ**。そして`no_connection`をtrueにします。このObservableはled_taskが購読していて、機体のLEDが「無線なし」の色に変わり、ピットの人間にも一目で分かります。指令の配布とまったく同じ経路（Observable）で「ゼロ」を配るので、下流のtaskたちにとってタイムアウトは特別なイベントではなく、ただの新しい指令です。**異常系を正常系と同じ配管で流す**——コードを増やさない、堅い作りです。

なお、rf_taskの待ち受けは`select3`で書かれていて、「受信完了の割り込み」「ロボットID変更」「周波数変更」の3つを同時に待ちます。周波数変更が飛び込めばSX1280を設定し直してから受信に戻る——SSLルールの「2キャリア切替」要件が、第9部で学んだselectそのものの形で実装されています。

## 50msという数字を考える

なぜ50msなのでしょうか。コードに理由のコメントはありませんが、両側から挟んで考えられます。

- **短くできない側**: 基地局は最大16台を1台ずつポーリングします。1台あたり送信＋応答窓（コード上は数ms）を使うので、自分の番が回ってくる間隔はチーム全体で数十msになり得ます。タイムアウトがそれより短いと、**異常がなくても毎回タイムアウト**してしまいます
- **長くできない側**: SSLのロボットは秒速数mで走ります。仮に2m/sなら、50msで10cm進みます。タイムアウトを500msにすれば、暴走時に1m突っ走ってから止まることになり、ルールのHalt猶予（2秒)や安全の観点で許容できません

つまり50msは「正常なポーリング周期では起きず、異常時の暴走距離を数十cmに抑える」あたりに置かれた値だと読めます。教材の最終プロジェクトではハートビート500ms・途絶判定2000msでした。あちらは途絶しても**warnを表示するだけ**の装置だから、ゆったりでよかった。こちらは**動く機械を止める**判定だから、桁が2つ厳しい。**タイムアウト値は「途絶の間に何が起き得るか」から逆算する**——両者を並べると、この原則がよく見えます。

| | 最終プロジェクト（C6×2） | このロボット |
|---|---|---|
| 定期通信 | ハートビート500ms | 基地局のポーリング（数十ms間隔） |
| 途絶判定 | 2000ms | **50ms** |
| 途絶時の動作 | warnログ＋表示 | **速度・キック・ドリブラー全ゼロ**＋LED表示 |
| 復帰 | 次のハートビートで自動復帰 | 受信再開で自動復帰（切断中は無期限待ち） |

## よくある誤解

- **「Wi-FiやBLEで作ればいいのに」** — BLE（Bluetooth Low Energy）を含むBluetoothはSSLルールで明確に禁止です（周波数を固定できないため）。Wi-Fiは禁止ではありませんが、周波数の調整可能性と混雑耐性・遅延の読みやすさから、多くのチームが専用トランシーバを選んでいます。無線の選定は性能表ではなく**運用環境とルール**から始まります
- **「タイムアウトで止まるのは通信機能の一部」** — このゼロ化は通信機能ではなく**安全機能**です。しかも1層目にすぎません。UARTキープアライブ、ハードウェアウォッチドッグ、ドリブラーのPWM途絶検出、基地局の物理ボタン——多層のフェイルセーフの全体像は10ページ目で扱います
- **「50msは万能の正解値」** — この値は「このポーリング設計・この機体速度」から逆算されたものです。あなたの装置が加湿器なら2000msで十分ですし、もっと速い機械なら50msでも長すぎます。持ち帰るべきは数字ではなく逆算の方法です

## 設計を考える

1. タイムアウト時の処理は「Observableにゼロを`set`する」だけで、モータを直接止めるコードはrf_taskにありません。この間接的な作りの利点を考えてください。

<details>
<summary>考え方の例</summary>

止め方の知識（減速のさせ方、モータ基板への伝え方）をrf_taskに持ち込まずに済みます。下流（UART橋のtask、モータ基板）は「速度指令ゼロ」を通常の指令として処理するので、異常系専用の経路やテストが不要です。また、ゼロ化の経路が正常系と同じなら、正常系が動いている限り異常系も動く——「めったに走らないコードが本番で初めて走って壊れる」事故を構造的に避けられます。弱点はrf_taskより下流が死んでいると効かないことですが、そこはUARTキープアライブとウォッチドッグという別の層が受け持ちます。

</details>

2. 切断中（no_connection == true）はタイムアウトなしの無期限待ちに切り替わります。もし切断中も50msタイムアウトのままだったら、何が起きるでしょうか。

<details>
<summary>考え方の例</summary>

すでに全出力ゼロなので安全上の問題はありませんが、50msごとに「タイムアウト→warnログ→ゼロを再set→受信再開」の空回りが続きます。ログは流れ続け、無線チップとアンプの再設定も繰り返されます。切断中にやるべきことは「次の1パケットをじっと待つ」だけなので、無期限待ちが最も無駄のない待ち方です。状態（接続中/切断中）によって最適な待ち方が変わる、小さいけれど気の利いた設計です。

</details>

## まとめ

- 無線はSX1280のFLRC 1.3Mb/s。宛先はロボット別32bit同期語でチップが振り分け、SKY66112で増幅する。TIGERs Mannheimの実績に倣った構成で、NRF24L01が本番で通信に失敗した2022年の教訓から選ばれた
- SSLルールは無線に申告制・2キャリア切替を課し、Bluetoothを禁止する。周波数変更はselect3の一分岐として実装され、ルールがそのままコードの形になっている
- 受信50ms途絶で速度・キック・ドリブラーをゼロにし、no_connectionを配る。教材のハートビート設計と同じ骨格を、動く機械の安全要件に合わせて2桁厳しくした実戦版
