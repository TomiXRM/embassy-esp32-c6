## このページでできるようになること

- RoboCup SSL（Small Size League）という競技の仕組みと制約を説明できる
- 題材のファームウェア（luhbots/luhsoccer_firmware）がどんなチームの、どんな実戦で使われたものか説明できる
- チームがRustを選んだ理由を、一次資料（TDP）に基づいて説明できる
- この応用編・全12ページの地図を手に入れる

## 先に結論

この応用編では、**国際大会で実際に戦ったサッカーロボットのファームウェア**を読みます。題材はドイツ・ハノーファーの学生チームluhbotsが公開しているluhsoccer_firmware（MITライセンス、Rust約17,000行）です。ロボット1台の中に**RP2040マイコンが3枚**、フィールド脇に**基地局がもう1枚**。合計4種類の基板が無線・UART・PWM・I2C・SPIでつながって、1つのチームとして動きます。キーボード編（応用編1）が「1台の中の並行処理」の総合演習だったのに対し、この応用編は「**複数のマイコンが協調する分散システム**」の総合演習です。教材の最終プロジェクトで作った「2台のESP32-C6の無線ボタン端末」は、実はこの世界の最小構成でした。ここからは同じ設計問題の実戦版を読みます。

## 身近なたとえ

RoboCup SSLは「**将棋のAI対戦の、駒が自走する版**」に近い競技です。人間は観客席にいるだけで、試合中は一切操作しません。天井のカメラが盤面（フィールド）全体を見て、フィールド脇のコンピュータ（AI）が「次の一手」を考え、無線で駒（ロボット）に指示を送ります。ロボットは指示どおりに走り、ボールを蹴ります。

——ただし実際の技術は、将棋よりずっと過酷です。将棋の駒は待っていてくれますが、ボールとロボットは秒速数メートルで動き続けます。「考えて、伝えて、動く」のループを毎秒何十回も回し続けなければならず、**通信が一瞬でも途切れたときにロボットをどう安全に止めるか**までがファームウェアの仕事です。

## RoboCup SSLとは — 公式ルールから

RoboCup SSL（Small Size League）は、ロボットサッカーの国際リーグの1つです。公式ルールから、この応用編の理解に必要な事実だけを押さえます。

| 項目 | ルール |
|---|---|
| チーム構成 | Division Aは11台、Division Bは6台 |
| ロボットの大きさ | **直径0.18m×高さ0.15mの円筒**に収まること |
| ボール | ゴルフボール（オレンジ色） |
| キック速度 | ボールを**6.5m/sより速く加速してはいけない** |
| 視覚 | 天井カメラの映像を**SSL-Vision**という共有ソフトが処理し、全チームに同じデータを配信 |
| 審判 | **Game Controller**というソフトが審判コマンドをネットワーク配信 |
| 自律性 | **完全自律**。試合中に人間がシステムへ入力することは禁止 |
| 緊急停止 | **Halt**コマンドが出たら全ロボット停止。制動の猶予は**2秒** |
| 無線 | 方式・出力・周波数を運営に申告。**2つの搬送波周波数から選べる**こと。**Bluetooth禁止**（周波数を固定できないため） |

つまり情報の流れはこうです: **共有カメラ（SSL-Vision）→ フィールド外のAIコンピュータ → 無線 → ロボット**。ロボット自身はフィールド全体が見えません。「目」も「頭脳」もフィールドの外にあり、ロボットは高速で正確な「手足」に徹します。だからこそ、無線リンクとロボット内部のファームウェアの品質が勝敗に直結します。

## 題材チーム — luhbots

luhbots Soccerは、ドイツのLeibniz Universität Hannover（ライプニッツ大学ハノーファー）の学生チームです。クラブとしてのluhbotsは2012年設立、SSLへの取り組みは2019年からです。

SSLでの戦績（公式結果より）:

- **2022年（バンコク）**: Division B **3位**（SSL初出場）
- **2023年（ボルドー）**: Division B **準優勝**（決勝でRobôCInに敗北。Ball Placementチャレンジ2位）
- **2024年（アイントホーフェン）**: Division Aに初参戦し**5位タイ**

初出場から2年でDivision B決勝、3年目に上位リーグへ——急成長したチームです。ただし正確に書いておくと、**luhbotsはSSLでは優勝していません**。「世界一」の実績は、同じクラブの**別部門**であるRoboCup@Work（産業ロボット部門）のもので、2015年（合肥）と2016年（ライプツィヒ）に世界チャンピオンになっています。サッカー部門と@Work部門は題材も別、実績も別です。混同しないでください。

この応用編で読むluhsoccer_firmwareは、上の2023年・2024年の機体で実際に使われた世代のファームウェアで、GitHubに**MITライセンス**で公開されています。ライセンスが明確なので、この教材では出典を明記した上でコードをそのまま引用します（キーボード編ではライセンス表記がなく引用を控えたのと対照的です）。

## なぜRustなのか — チーム自身の言葉

チームは毎年、設計を説明するTDP（Team Description Paper）を公開しています。2023年のTDPに、ファームウェアをRustで書いた理由がそのまま書かれています。

> With the change of the microcontroller the firmware is rewritten entirely. In foresight of possible future microcontroller upgrades, the firmware is coded in Rust which would allow the code to be updated accordingly, instead of completely rewritten. The firmware uses the embedded-hal crate to allow for reuse of code across multiple architectures.
>
> （要旨訳: マイコンの変更に伴い、ファームウェアは全面的に書き直した。将来またマイコンを更新する可能性を見越して、そのときに全書き直しではなく更新で済むよう、Rustで記述した。embedded-halクレートを使うことで、複数のアーキテクチャ間でコードを再利用できるようにしている。）
>
> — luhbots Soccer, Team Description for RoboCup 2023, Section 4

注目してほしいのは、理由が「速いから」でも「流行っているから」でもなく、「**マイコンを乗り換えてもコードが生き残るから**」だという点です。embedded-halという共通の型に依存して書けば、チップ固有のHALを差し替えるだけで大部分のコードが再利用できる——第11部で学んだ「依存の向きの設計」を、チームは実戦の武器として使っています。実際このファームウェアは、Cortex-M0+（RP2040）とCortex-M4（ATSAM4E）という**別アーキテクチャのマイコンが混在**するのに、共通ライブラリ群を共有しています。

なお、TDPには「Embassy」という語は出てきません。EmbassyをRP2040側で使っている事実は、リポジトリのCargo.toml（embassy-rp、embassy-executor、embassy-time、embassy-sync）で確認できます。これは「Embassy on RP2040」の話であり、この教材のESP32-C6でそのまま動くコードではないことも、キーボード編と同じです。

そしてもう1つ、この応用編ならではの視点があります。実戦投入されたコードには、**教科書に載らない現実**が写っています。空のREADME、コメントアウトの残骸、「基板の設計ミスをファームウェアで回避した」形跡。それらも隠さずに読みます。完璧ではないコードが世界2位まで戦えた——その足腰がどこにあるのかを見るのが、この応用編の楽しみ方です。

## よくある誤解

- **「サッカーロボットのAI（戦略）を読むのだ」** — 違います。戦略はフィールド外のコンピュータ上のソフトウェアで、この応用編の範囲外です。読むのはロボットと基地局の**ファームウェア**、つまり「指示を受けて、正確に動き、安全に止まる」部分です
- **「優勝チームのコードだから正解集だ」** — luhbotsはSSLで優勝していませんし（最高はDivision B準優勝）、コードにも未完成な部分が残っています。「正解集」ではなく「実戦に耐えた実例」として、良い判断も妥協も両方読み取ります
- **「RP2040とATSAM4Eの話だからC6には関係ない」** — キーボード編と同じで、逆です。チップ非依存の設計（task分割、メッセージ定義、フェイルセーフ）こそが持ち帰る対象で、この応用編の各ページで毎回「C6ならどうなるか」に触れます

## 設計を考える

1. SSLのルールでは「目」（カメラ）と「頭脳」（AI）がフィールドの外にあり、ロボットには無線で指示が届きます。この構図のとき、ロボット側のファームウェアが**絶対に自力でやらなければいけない仕事**を1つ挙げてください。

<details>
<summary>考え方の例</summary>

「無線が途切れたときに安全に止まること」です。指示が来ない間、ロボットは外部の頭脳から切り離されています。最後の指示のまま走り続ければ、人やほかのロボットにぶつかります。判断できない時間は「何もしない」側へ倒す——教材の最終プロジェクトでハートビート途絶を検出したのと同じ発想が、ここでは安全要件になります（4ページ目で実物のコードを読みます）。

</details>

2. チームがTDPで挙げたRust採用の理由は「将来のマイコン更新に備えて」でした。実際、このチームは2022年から2023年にかけてマイコン構成を大きく変えています。もしファームウェアが特定チップのレジスタ操作だらけのC言語で書かれていたら、何が起きたと考えられますか。

<details>
<summary>考え方の例</summary>

TDPに書かれているとおり「完全な書き直し」です。レジスタ操作はチップ固有なので、チップが変われば通信・モータ制御・センサ読み取りのすべてに手が入ります。embedded-halの型に依存して書いてあれば、書き直しは「HAL実装の差し替え」に局所化されます。第11部で学んだ「依存の向き」が、開発期間が限られる学生チームにとって死活問題になる実例です。

</details>

## まとめ

- 題材はRoboCup SSLで実戦投入されたluhbotsのRustファームウェア（MITライセンス、約17,000行）。カメラとAIはフィールド外、ロボットは無線で動く「手足」という競技構造がすべての設計を規定している
- luhbotsのSSL最高成績は2023年Division B準優勝。優勝歴は同クラブの@Work部門（2015・2016）のもので、別部門の実績
- Rust採用の理由は一次資料（2023 TDP)に明記されている: マイコン更新に耐えるコードを、embedded-halによる抽象化で実現するため
